## Summary

The PROPERTIES document specifies 36 testable system properties across 9 domains for Phase 2.
The overall structure is clear and the priorities are sensible. However, there are **7 findings** of varying severity that must be addressed before implementation starts: two are blocking (PROP-NORTH-03 altitude mismatch against the TSPEC-pinned vector; and PROP-SCHEMA-01 `id` type conflict), three are moderate (test class name mismatches that would cause build failures; interference-property method names that do not exist on the canonical interface; PROP-LOCATION-03 boundary uses 1 s granularity where TSPEC specifies 1 ms), and two are informational (PROP-INTERFERENCE-03's fallback constants are asserted without spec grounding; PROP-PERSIST-02's force-stop mechanism diverges from the TSPEC approach without explanation).

## Findings

| ID | Severity | Property | Finding | Recommendation |
|----|----------|---------|---------|----------------|
| SE-P-01 | **Critical** | PROP-NORTH-03 | **Altitude mismatch in pinned test vector.** PROP-NORTH-03 specifies `altM=1600.0` for the NOAA-pinned test vector. TSPEC §10.1 `Wmm2025ModelTest` specifies the same pinned vector at `altM=0.0`. A 1600 m altitude difference shifts the WMM declination result (typically by ~0.01°–0.05°). With a tolerance of only ±0.1°, a mismatched altitude could put the WMM math test at risk of a spurious failure or, worse, mask a real computation error if the developer codes the test using the PROPERTIES altitude. Both documents must specify the same value; the TSPEC-canonical altitude is `altM=0.0`. | Change PROP-NORTH-03 altitude to `altM=0.0` to match the TSPEC §10.1 `Wmm2025ModelTest` canonical vector (`lat=40.0, lon=−105.0, altM=0.0, epochYear=2025.5`). Update the `When` clause in PROP-NORTH-01 accordingly if needed (PROP-NORTH-01 references the same location with `altM=1600.0`). |
| SE-P-02 | **Critical** | PROP-SCHEMA-01 | **`id` field type is `Long` (auto-increment) in PROPERTIES but `String` (UUID v4) in FSPEC §6.1 and TSPEC §4.1.** PROP-SCHEMA-01's type table lists `id` as `Long` (autogenerated primary key). FSPEC §6.1 specifies `id: String, TEXT NOT NULL, Primary key, UUID v4`. TSPEC §4.1 defines `val id: String` in `BearingRecord`. A reflection-based test using the PROPERTIES type table would assert `id.returnType.classifier == Long::class` and fail against the correct implementation — or, if a developer naively implements `id` as `Long`, it would conflict with all other specs. | Change PROP-SCHEMA-01 `id` row to `String` (UUID v4). The `Long` type almost certainly persists from an earlier draft before the UUID decision was made. |
| SE-P-03 | **High** | PROP-INTERFERENCE-01, PROP-INTERFERENCE-02 | **Method names `getFieldStrength_uT()` and `getInclination_deg()` do not exist on `MagneticFieldModel`.** The PROPERTIES test bodies reference `MagneticFieldModel.getFieldStrength_uT(...)` (PROP-INTERFERENCE-01) and `MagneticFieldModel.getInclination_deg(...)` (PROP-INTERFERENCE-02). The canonical interface defined in TSPEC §2.1 and FSPEC §6.3 uses `getExpectedFieldMagnitude(latDeg, lonDeg, altM, epochYears): Float` and `getExpectedInclination(latDeg, lonDeg, altM, epochYears): Float`. Tests coded verbatim from the PROPERTIES spec would not compile. | Replace `getFieldStrength_uT()` with `getExpectedFieldMagnitude()` and `getInclination_deg()` with `getExpectedInclination()` throughout PROP-INTERFERENCE-01 and PROP-INTERFERENCE-02, using the TSPEC §2.1 parameter names (`latDeg`, `lonDeg`, `altM`, `epochYears`). |
| SE-P-04 | **High** | Multiple (see below) | **Test class names in PROPERTIES do not match TSPEC §10 test classes.** The following PROPERTIES test classes have no matching entry in TSPEC §10 and would require the developer to invent new test classes: `TrueNorthHeadingTest` (PROP-NORTH-01/02), `Wmm2025AccuracyTest` (PROP-NORTH-03), `Wmm2025FallbackTest` (PROP-NORTH-04), `NorthToggleUiTest` (PROP-NORTH-05/06), `NorthTypeToggleTest` (PROP-DECL-01), `DeclinationInfoPanelTest` (PROP-DECL-02), `OfflineDeclinationTest` (PROP-DECL-03), `LocationResolverTest` (PROP-LOCATION-01), `LocationLifecycleTest` (PROP-LOCATION-02), `LocationCacheExpiryTest` (PROP-LOCATION-03), `LocationPermissionDeniedTest` (PROP-LOCATION-04), `BearingCaptureTimestampTest`, `BearingInterferenceFlagTest`, `BearingCalibrationVersionTest`, `BearingNorthTypeTest`, `BearingSchemaTest`, `BearingNotesCoercionTest`, `BearingDoubleTapTest`, `BearingCaptureButtonStateTest`, `InterferenceWmmBaselineTest`, `InterferenceFallbackTest`, `ExtremeLatitudeAdvisoryTest`, `ExtremeLatitudeConfidenceTest`, `ColdStartBenchmark`, `WarmStartBenchmark`. A PROPERTIES document that references non-TSPEC class names creates an inconsistency that could result in duplicated or orphaned test classes during implementation. | Align all PROPERTIES `Test Class` column entries with TSPEC §10 class names. Where a property's assertions span multiple TSPEC classes (e.g., `BearingCaptureUseCaseTest` + `CompassViewModelTest`), list both. Where no exact TSPEC class exists, raise a gap against the TSPEC rather than silently defining a new class. |
| SE-P-05 | **Moderate** | PROP-LOCATION-03 | **Boundary condition granularity is 1 second but TSPEC specifies 1 millisecond.** PROP-LOCATION-03 specifies the expired boundary as `30 * 86_400_000L + 1_000L` ms (1 s over 30 days). TSPEC §10.1 `LocationRepositoryTest` specifies `false` when age = 30 days exactly and `true` when age = 30 days + 1 ms. The FSPEC §2.4 rule is strict: "one from 30 days + 1 second is expired." Both the FSPEC and TSPEC agree: the check is `elapsed_ms < 30 * 86_400_000L` (strictly less than), so a 1 ms overshoot already triggers expiry. Using 1 s in the PROPERTIES test is not wrong, but it misses the canonical 1 ms boundary case that reveals off-by-one errors in `<` vs `<=` implementations. | Add a second sub-case to PROP-LOCATION-03: `nowMs = T + 30 * 86_400_000L + 1L` (1 ms over) must return `true`. The 1 s sub-case may stay as an additional confirming case. |
| SE-P-06 | **Moderate** | PROP-INTERFERENCE-03 | **Fallback constants (50 µT, 45°) are asserted as exact values but are not grounded in any FSPEC or TSPEC normative text.** PROP-INTERFERENCE-03 asserts `expectedMagnitude_uT == 50.0f` and `expectedInclination_deg == 45.0f` as the Phase 1 EMA fallback when no location is available. Neither value appears in FSPEC §2.6 or TSPEC §3.8 as normative constants; FSPEC §2.6 step 4c references "50 µT global-average constant if the EMA has not yet stabilized" but does not pin the inclination default. TSPEC §3.8 states the EMA is retained as a no-location fallback but does not assign a fixed constant. If the implementation uses a different EMA seed or inclination default, a passing implementation would break this test. | Change the assertion to a behavioral test: verify that when no location is available and the EMA is used, `InterferenceDetector` does not call `MagneticFieldModel.getExpectedFieldMagnitude()` and does not throw. Alternatively, raise a TSPEC issue to normatively specify the EMA seed values so the constant assertions become legitimate. |
| SE-P-07 | **Low** | PROP-PERSIST-02 | **`Process.killProcess()` mechanism for force-stop diverges from TSPEC `BearingForceStopTest`.** PROP-PERSIST-02 specifies `Process.killProcess()` injected to fire during the Room `@Transaction` write. TSPEC §10.2 `BearingForceStopTest` explicitly uses `ProcessPhoenix` (already in androidTest dependencies) to kill and restart the process. Injecting `Process.killProcess()` mid-transaction from inside the test process is not reliably achievable without root access, and differs from the `adb shell am force-stop` or `ProcessPhoenix` approach the TSPEC specifies. The PROPERTIES approach is untestable as written on a standard CI emulator. | Replace the `Process.killProcess()` mechanism with `ProcessPhoenix.triggerRebirth(context)` per TSPEC §10.2 `BearingForceStopTest`. This is implementable without root access and is already in the androidTest dependency set. |

## Phase 1 Regression Section Assessment

The Phase 1 regression section (bottom of the document) is **complete and well-structured**. It correctly identifies all seven at-risk Phase 1 domains (PROP-FUSION, PROP-DETECT, PROP-CONF, PROP-PERSIST, PROP-DISPLAY, PROP-CAL, PROP-NFR) and provides explicit regression guard strategies for each. The explicit CI re-run list matches the actual risk profile introduced by Phase 2 changes to `InterferenceDetector` and `ConfidenceModel`.

## PROP-PERSIST-01 Implementability Assessment

**Implementable without root access.** The test reads `context.getDatabasePath("bearing_records.db")` as raw bytes inside an `androidTest` instrumented test. The test app has file-system access to its own database path without elevated privileges. The magic-byte check is the same pattern used in Phase 1 and is confirmed implementable in TSPEC §10.2 `BearingEncryptionTest`. No findings.

## PROP-PERSIST-02 Force-Stop Implementability

**Partially untestable as written — see SE-P-07.** The `Process.killProcess()` injection approach would require the test harness to interrupt a suspend function in-flight inside the Room transaction, which cannot be done deterministically from within the same JVM process without root or debug instrumentation. `ProcessPhoenix` (per TSPEC) is the correct mechanism and is already available.

## PROP-NORTH-03 WMM Vector Pinning Assessment

After correcting the `altM` discrepancy (SE-P-01), the pinning approach is sound. The vector `lat=40.0°N, lon=−105.0°W, altM=0.0, epochYear=2025.5` is documented as the NOAA WMM2025 reference vector in TSPEC §10.1, and the tolerance of ±0.1° is correctly tight for a pre-committed reference value. The assertion structure (single pinned vector, supplementary vectors optional) is consistent with TSPEC guidance.

## PROP-INTERFERENCE-01/02/03 Consistency with Phase 1 EMA Fallback

PROP-INTERFERENCE-01 and PROP-INTERFERENCE-02 correctly model the Phase 2 architecture: the caller (`CompassViewModel`) now passes WMM values as `metrics.expectedField_uT` and `metrics.expectedInclination_deg`, consistent with TSPEC §3.8. PROP-INTERFERENCE-03 captures the no-location EMA fallback correctly in intent, but the hardcoded constant assertions are ungrounded (SE-P-06). Structural consistency with the TSPEC Phase 1 EMA design is confirmed.

## Recommendation: Need Attention

Two critical type/value discrepancies (SE-P-01, SE-P-02), one compile-blocking method name mismatch (SE-P-03), and one systemic test class alignment gap (SE-P-04) must be resolved before implementation. SE-P-05, SE-P-06, and SE-P-07 should be addressed but are not blockers to starting implementation. Re-review recommended after SE-P-01 through SE-P-04 are addressed.
