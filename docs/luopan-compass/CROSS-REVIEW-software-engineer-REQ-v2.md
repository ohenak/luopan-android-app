## New Findings

| ID | Severity | Section | Finding | Recommendation |
|----|----------|---------|---------|----------------|
| N-01 | High | §5.3.1; §10.2 | **`calibration_schema_version` field does not exist — migration DDL scope is internally contradictory.** §5.3.1 defines `calibration_version` as resolving to `CalibrationRecord.calibration_schema_version` (currently `1`). Inspecting `CalibrationRecord.kt`, the entity has no such field — it carries only `id: Int`, `recorded_at: Long`, `hard_iron_x/y/z: Float`, `soft_iron_*: Float`, and `quality: String`. Adding `calibration_schema_version` to `CalibrationRecord` requires adding a column to the existing `calibration_records` table (`ALTER TABLE calibration_records ADD COLUMN calibration_schema_version INTEGER NOT NULL DEFAULT 1`). §10.2 item 2 explicitly states that `Migration(1, 2)` "MUST execute only `CREATE TABLE IF NOT EXISTS bearing_records (...)` — it MUST NOT drop or alter `calibration_records`". These two specifications are directly contradictory. Engineering cannot satisfy both simultaneously. One of three resolutions is needed: (a) add the `ALTER TABLE` statement to the migration and update §10.2 item 2 accordingly; (b) redefine `calibration_version` to use an existing field (`CalibrationRecord.id` is always 1 for the current record, or `recorded_at` as a timestamp); or (c) add a static constant `CALIBRATION_SCHEMA_VERSION = 1` to the app layer (not stored in `CalibrationRecord` at all) and document that it changes only when the entity schema changes. Option (c) avoids the migration inconsistency at the cost of making the version implicit. | Decide among options (a), (b), or (c) and update both §5.3.1 and §10.2 item 2 to be consistent. If option (a): list the `ALTER TABLE` DDL explicitly in §10.2. If option (b) or (c): remove `calibration_schema_version` from `CalibrationRecord` field table entirely and restate what the integer value represents. |
| N-02 | Medium | §5.1 (MagneticFieldModel interface) | **`epochYears` computation contract unspecified.** All four methods on `MagneticFieldModel` accept `epochYears: Double` (e.g., `2025.5` for 1 July 2025), but the REQ never specifies how this value is computed from a wall-clock date. The WMM-standard formula is `year + (dayOfYear − 1) / daysInYear` evaluated in UTC. Without this contract: (1) two engineers calling the interface may use different formulas (e.g., using local timezone vs. UTC, or `dayOfYear / 366` vs. the fractional formula); (2) the static test vector in Scenario B states epoch `2025.5` = 1 July 2025, which is correct under the standard formula — but without the formula being normative, the test vector cannot be reproduced mechanically from a `java.time.LocalDate`. | Add a one-sentence normative definition of `epochYears` in §5.1 immediately after the interface block: "Epoch years MUST be computed as `year + (dayOfYear − 1.0) / daysInYear` using UTC date, where `daysInYear` is 365 or 366 for the calendar year." |
| N-03 | Medium | §10.2 | **Room schema export is disabled (`exportSchema = false`), conflicting with `MigrationTestHelper` requirements.** `LuopanDatabase` is currently declared `@Database(..., exportSchema = false)`. §10.2 item 5 requires an instrumented `MigrationTest` using Room's `MigrationTestHelper`. `MigrationTestHelper` requires exported JSON schema files to reconstruct the baseline database state for the migration-under-test — it will fail at runtime if schema export is disabled. The REQ specifies the testing obligation but does not require enabling schema export, leaving a gap that will surface only during TSPEC implementation. | §10.2 should add an explicit requirement: "`exportSchema` MUST be set to `true` in the `@Database` annotation, and the generated schema JSON files MUST be committed to source control in `app/schemas/` (or equivalent) so that `MigrationTestHelper` can reconstruct pre-migration database states." |
| N-04 | Medium | §8 Scenario G | **"MAY appear" banner is untestable as written.** Scenario G's pass criteria do not mention the informational banner, but the scenario body states "a one-time informational banner MAY appear explaining that location was denied." The word "MAY" creates ambiguity: it is neither pass nor fail. A TE writing an instrumented test cannot assert on "MAY" behavior — the test must either assert the banner appears (asserting on a MAY is incorrect) or assert it does not appear (equally wrong). This ambiguity will surface as a TE question during TSPEC authoring. | Replace "MAY appear" with a deterministic contract: either "MUST appear on the first denied session" (then add it to the pass criteria) or "is optional and not tested in Phase 2." If the banner is intentional UX, specify the `SharedPreferences` key used to track "banner shown once" so engineers don't duplicate or conflict with the REQ-CAPTURE-06 privacy dialog state. |
| N-05 | Low | §5.3.1 (`display_mode` field) | **`display_mode` enum includes values (`LUOPAN`, `SIGHTING`) that are unreachable in Phase 2.** `BearingRecord.display_mode` is typed as `Enum: MODERN, LUOPAN, SIGHTING`. In Phase 2, only `MODERN` mode exists — `LUOPAN` (Phase 3) and `SIGHTING` (Phase 5) cannot be the active display mode. The REQ does not specify what Phase 2 implementations MUST write for this field, leaving an implicit assumption that all Phase 2 records will carry `MODERN`. If the TSPEC does not normalize this, Phase 4 history queries that group by `display_mode` may encounter unexpected `null` or incorrect values. | Add a note to §5.3.1: "In Phase 2, `display_mode` MUST always be written as `MODERN` for all captured records, as `LUOPAN` and `SIGHTING` modes are not yet implemented. Phase 3 and Phase 5 will write the respective values." |
| N-06 | Low | §8 Scenario B (static WMM test vector) | **Static test vector lacks a traceable citation.** Scenario B provides a static WMM2025 test vector (40.0°N, −105.0°W, 0 m, epoch 2025.5 → declination +8.93°E ±0.1°, inclination +66.0° ±0.5°, total field 52,300 nT ±200 nT) and states it is "derived from the WMM2025 Technical Note (NOAA, 2024)" without a table number, page, or URL. If the test vector is wrong (e.g., because NOAA published a correction or the epoch date was misread), there is no audit trail to detect the error. The REQ itself says "If NOAA publishes a corrected value before Phase 2 ships, the PM and SE must update this table" — implying NOAA correctness is the authoritative source, not this table. | Add the specific section/table reference from the WMM2025 Technical Note (e.g., "WMM2025 Technical Note, Section 4, Table 3, NOAA, December 2024, https://www.ngdc.noaa.gov/geomag/WMM/…") so that future reviewers can independently verify the values. If no official test vector table exists for exactly this location, note that the values were computed via the NOAA online calculator at a specific date and record the calculator URL. |

## Additional Observations (Informational, No Action Required)

**F-07 (Scenario A 200 ms criterion):** The PM's v0.2-draft adds a detailed measurement definition to Scenario A (Espresso `IdlingResource` or Macrobenchmark frame timing, tolerance ±50 ms, reference device class per REQ-NFR-08). This fully addresses the latency ambiguity raised in F-07. No finding raised.

**F-08 (BearingRecord database sharing):** §5.3.1 and §10.2 implicitly confirm that `BearingRecord` is added to the same `@Database` as `CalibrationRecord` (schema version bump from v1 to v2 on the same `luopan.db`). This is the correct implementation choice and is now sufficiently specified for the TSPEC. No finding raised.

**F-09 (Main-UI toggle vs. `declinationMode` setting):** The v0.2-draft does not explicitly resolve how the main-UI True N / Magnetic N toggle maps to the existing `SettingsRepository.declinationMode` string constants (`auto`, `manual`, `magnetic`). Setting `declinationMode = "auto"` is the natural implementation for True N, but this is not stated. This ambiguity remains, though it is a TSPEC-level concern rather than a REQ-level blocker — the SE authoring the TSPEC should resolve it. Not raised as a new finding at this stage, but noted for TSPEC review.

**F-11 (WMM coefficient count):** The REQ still states "168 Gauss coefficients." The WMM2025 `.COF` file contains both main-field coefficients (g, h) and secular variation coefficients (ġ, ḣ) — 168 each, 336 total. Without secular variation coefficients, declination accuracy degrades by the epoch secular rate across the 2025–2030 validity window (~0.1–0.3°/year at mid-latitudes). This is acceptable for the declared ±0.1° test vector tolerance only at epoch 2025.5; at 2029.5 the error would exceed ±0.5°. The REQ's stated claim that 168 coefficients suffice is technically undersimplified, but fixing it is deferred to the TSPEC where the WMM2025 model implementation is specified. Not raised as a new finding but the TSPEC author should address SV coefficient inclusion.

**§10.3 `testFixtures` source set:** The requirement that `FakeMagneticFieldModel` reside in the `testFixtures` source set requires the `java-test-fixtures` Gradle plugin, which is not currently in the build. This is a build configuration change that the TSPEC must specify — it is not a REQ issue.

## Recommendation: Approved with Minor Issues

The PM has substantively resolved the four High findings from Iteration 1. The `MagneticFieldModel` interface contract (F-01), location permission specification (F-02), migration safety requirement (F-03), and `interference_flag` semantics (F-10) are now adequately specified for TSPEC authoring.

One High finding remains (N-01): the `calibration_schema_version` contradiction between §5.3.1 and §10.2 item 2 must be resolved before the TSPEC can write the migration DDL. This is a targeted fix — the PM needs to choose one of three defined options and update two sections. It does not require a broad document revision.

The Medium findings (N-02, N-03, N-04) are TSPEC-unblocking issues that should be addressed in a v0.3-draft or incorporated as normative notes in the TSPEC itself. N-01 is the only item that blocks TSPEC authoring.

**Recommended action:** PM to resolve N-01 (targeted fix), address N-02 through N-04 as inline notes, and proceed to TSPEC authoring. A third SE review iteration is not required provided N-01 is resolved.
