# Cross-Review: SE — FSPEC-luopan-p2-true-north-capture

**Reviewer role:** Senior Software Engineer
**Document reviewed:** FSPEC-luopan-p2-true-north-capture.md
**Date:** 2026-04-25
**Iteration:** 1

---

## Summary

The FSPEC is well-structured and covers the seven functional flows with useful state diagrams and 35 acceptance tests. However, it contains six direct contradictions with the approved REQ v0.2-draft on schema field types and semantics — the most serious being `calibration_version`, which the FSPEC redefines as a WMM model name string while the REQ defines it as an integer schema-version counter. The permission-rationale flow explicitly contradicts a MUST clause in the approved REQ. Additionally, the FSPEC introduces a new `Clock` interface that duplicates the existing `TimeSource` already in the codebase, mischaracterises the Phase 1 interference baseline implementation, and leaves the GPS location update interval completely unspecified. These issues, if unresolved before TSPEC authoring, will produce an inconsistent implementation.

## Findings

| ID | Severity | Section | Finding | Recommendation |
|----|----------|---------|---------|----------------|
| F-01 | High | §6.1 — BearingRecord / BR-02 | **`calibration_version` type and semantics directly contradict REQ v0.2 §5.3.1.** FSPEC defines the field as `String` (values `"WMM2025"` or `"AndroidGeoField"`) recording the active magnetic model. The approved REQ defines it as `Integer` (resolves to `CalibrationRecord.calibration_schema_version`, currently `1`), recording the Room schema version of the calibration record used at capture time. These are different things stored in the same column. Implementation will be split between two interpretations. One is correct; the other must be removed. Engineers cannot implement both. | PM and SE must align on a single definition before TSPEC is drafted. The two interpretations serve different purposes: the model name supports auditability of the WMM algorithm used; the schema integer supports future DB migration tracking. Both may be desirable, but they need distinct field names if both are kept. |
| F-02 | High | §2.4 — GPS, step 2 | **Permission rationale flow directly contradicts REQ v0.2 §10.1.** FSPEC states "the app does NOT show a pre-permission rationale dialog in Phase 2 — the permission is requested directly." The approved REQ states "A rationale dialog MUST be shown before the system permission prompt if `shouldShowRequestPermissionRationale()` returns `true`." Skipping the rationale when the OS signals it is required is an Android UX best-practice violation and will cause tests that check for `shouldShowRequestPermissionRationale()` handling to fail. | The FSPEC must be corrected to match the REQ MUST clause. Add a step: "If `shouldShowRequestPermissionRationale()` returns `true`, show a brief in-app explanation dialog before launching the system permission prompt." |
| F-03 | High | §2.6 — FSPEC-DETECTUPGRADE, step 1 and step 4 | **Phase 1 runtime interference baseline is mischaracterised as `GeomagneticField`.** The FSPEC states "In Phase 1, the expected field magnitude was derived from Android's `GeomagneticField` class." This is factually incorrect. The live Phase 1 `CompassViewModel` computes `baselineFieldUt` via a 99/1 exponential moving average of the measured sensor field — `GeomagneticField` is used only at calibration time (to compute the target sphere radius in `CalibrationEngine.getTargetRadius`), not at runtime. The FSPEC's fallback specification — "fall back to `GeomagneticField` for the expected values (the Phase 1 behavior)" — consequently describes a fallback that does not match any current code path. | Correct step 1 to accurately describe the Phase 1 baseline (sensor EMA). Restate the fallback hierarchy explicitly: (a) WMM2025 with resolved location, (b) `GeomagneticField` with resolved location when WMM2025 is expired or fails, (c) sensor EMA (current Phase 1 behavior) or 50 µT constant when no location is available. Note that `GeomagneticField` itself requires lat/lon/alt/time and cannot serve as a "no-location" fallback. |
| F-04 | High | §6.1 — BearingRecord schema | **Multiple field-level type conflicts with REQ v0.2 §5.3.1.** Three fields in the FSPEC schema table differ from the REQ definitions: (a) `notes` max length: FSPEC §2.5 step 6 and §6.1 both say 500 characters; REQ §5.3.1 says 1000 characters. (b) `bearing_deg` Kotlin type: FSPEC says `Double`; REQ says `Float (0.000–360.000)`. (c) `lat`/`lon` Kotlin types: FSPEC says `Double?`; REQ says `Float`. Additionally, `captured_at` type differs: FSPEC says `Long` (epoch milliseconds); REQ says `ISO 8601 UTC timestamp` (implying a string). These conflicts mean the Room entity cannot be correctly defined without resolving them. | Align all field definitions with the approved REQ before TSPEC authoring. Recommendation on substance: use `Double` for `lat`/`lon` (6 decimal places ≈ 11 cm precision, suitable for WMM inputs; `Float` gives only ~11 km precision); store `captured_at` as `Long` (epoch ms) with the ISO 8601 conversion at the display layer; clarify `bearing_deg` type and `notes` max length with PM as a single errata correction to the REQ. |
| F-05 | Medium | §6.2 — Clock Interface | **`Clock` duplicates the existing `TimeSource` interface in `com.luopan.compass.sensor`.** The FSPEC introduces a new `Clock` interface with a single `nowMs(): Long` method in `com.luopan.compass.util`. The codebase already has `TimeSource` in `com.luopan.compass.sensor` which exposes `nowMs(): Long` and `nowNs(): Long`. Introducing a second time-abstraction interface will result in two parallel test doubles (`FakeClock` and the existing injected `TimeSource` in `SensorRateMonitor`) and increase cognitive overhead. Additionally, the FSPEC names the production implementation `SystemClock`, which collides with `android.os.SystemClock` — both would be in scope in any file that imports both, causing a naming collision. | Replace `Clock` with `TimeSource` throughout the FSPEC. The `FakeClock` test double already exists conceptually as a `FakeTimeSource`. If a `nowNs()` method is not needed by the new Phase 2 code paths, the TSPEC can note that only `nowMs()` is called; no new interface is required. Rename the production implementation to `SystemTimeSource` (which already exists in the codebase). |
| F-06 | Medium | §6.3 — MagneticFieldModel Interface | **FSPEC describes `MagneticFieldModel` as "the existing interface pattern" but the interface does not exist in the codebase.** No `MagneticFieldModel` interface has been implemented in Phase 1; the codebase has only `LocationProvider` and `TimeSource` as injected protocols. The REQ v0.2 §5.1 defines the full Kotlin interface signature with three method signatures; §6.3 of the FSPEC only provides a return-value table without the method signatures. An engineer reading §6.3 alone cannot implement the interface without consulting the REQ. | Either reproduce the full interface definition from REQ §5.1 in §6.3, or add a cross-reference: "See REQ §5.1 for the required Kotlin interface signature." Remove the word "existing" — the interface must be created in Phase 2. |
| F-07 | Medium | §2.4 — GPS, step 9 | **GPS `requestLocationUpdates` parameters are unspecified.** The FSPEC states "periodic GPS updates (low-power mode, not high-accuracy continuous mode)" but does not specify the `minTime` interval, `minDistance` threshold, or which `LocationManager` provider to use (GPS_PROVIDER, NETWORK_PROVIDER, or a combined strategy). These parameters directly affect battery consumption (REQ-NFR-03) and the freshness of the cached location. Without a specification, two engineers will make different choices, producing inconsistent battery characteristics across builds. | Add a new step or business rule specifying: provider (recommended: `LocationManager.GPS_PROVIDER` when permission granted), `minTimeMs` (recommended: 60 000 ms / 1 minute), and `minDistanceM` (recommended: 100 m). These numbers should be validated against REQ-NFR-03 battery target. Alternatively, defer these parameters to the TSPEC but note them here as "to be specified by engineering." |
| F-08 | Medium | §2.4 — GPS, permission flow | **`ACCESS_COARSE_LOCATION` is absent from the permission request flow.** FSPEC §2.4 step 2 discusses only `ACCESS_FINE_LOCATION`. REQ v0.2 §10.1 requires both `ACCESS_FINE_LOCATION` and `ACCESS_COARSE_LOCATION` to be requested in the same `requestPermissions()` call, and mandates that both are declared in `AndroidManifest.xml` for API 26–30 compatibility. The FSPEC business rules (§2.4 last bullet) correctly state "ACCESS_COARSE_LOCATION is not sufficient" but the permission request flow never includes it. | Update step 2 of the permission request flow to: "request both `ACCESS_FINE_LOCATION` and `ACCESS_COARSE_LOCATION` in a single `requestPermissions` call, per REQ §10.1." |
| F-09 | Medium | §1.3 — Preconditions | **Cross-reference to TSPEC Phase 1 §10.2 is incorrect.** The precondition note states "see §10.2 of TSPEC Phase 1" for the Room migration context. TSPEC Phase 1 §10.2 covers the heading update rate requirement (REQ-NFR-02), not the database schema. The migration information is in TSPEC Phase 1 §9 (Persistence Layer). | Change the reference from "§10.2" to "§9" of TSPEC Phase 1. |
| F-10 | Medium | §2.4 — GPS, step 1 | **Lifecycle trigger is `onResume` but Phase 1 architecture uses `onStart`/`onStop`.** FSPEC step 1 states "On foreground entry (`Activity.onResume`)..." The existing `CompassActivity` registers and unregisters sensors in `onStart`/`onStop`, not `onResume`/`onPause`. The TSPEC Phase 1 §2.1 explicitly motivates `onStop` over `onPause` to avoid false interruptions during permission dialogs. If location resolution runs in `onResume` while sensors run in `onStart`, there will be a window where sensors are active but location is not resolved. The lifecycle callback must be consistent. | Change "Activity.onResume" to "Activity.onStart" throughout §2.4 to match the established Phase 1 lifecycle pattern. |
| F-11 | Medium | §7 — Acceptance Tests | **REQ-NFR-08 cold-start performance (≤3 s warm / ≤5 s cold) has no acceptance test.** The 35 tests cover all functional scenarios but no test covers the cold-start or warm-cache latency target defined in REQ-NFR-08. The REQ defines a Macrobenchmark measurement method with specific `StartupMode.COLD` and `StartupMode.WARM` parameters. Without an AT, this P1 requirement has no verification gate. | Add an AT-H group with at least two tests: AT-H-01 (cold start: `Application.onCreate()` to first non-STABILIZING heading ≤5 s, Macrobenchmark `StartupMode.COLD`, ≥5 iterations, median); AT-H-02 (warm start: same gate ≤3 s, `StartupMode.WARM`). Reference the measurement method from REQ §5.4. |
| F-12 | Medium | AT-A-01 | **AT-A-01 does not specify the measurement method for the 200 ms budget.** The assertion states "transition completes in ≤200 ms" but gives no measurement mechanism. The approved REQ Scenario A provides this: "wall-clock time from `View.performClick()` on the toggle to the first frame where the heading `TextView` displays a value differing from the pre-tap value by ≥ (`declination − 0.5°`), measured via Espresso `IdlingResource` or Macrobenchmark frame timing; tolerance ±50 ms." Without this, the test is not unambiguously implementable. | Reproduce the REQ measurement definition in the AT-A-01 Precise Assertion cell. |
| F-13 | Medium | AT-C-02 | **AT-C-02 uses `±1 day tolerance` for the cache age display but the REQ specifies exact boundary semantics.** The FSPEC allows the displayed age to differ by up to a full day from the expected value. REQ v0.2 Scenario C states the displayed string MUST equal `"Using last known location (15 days ago)"` when the cache age is exactly 15 days, and that the days boundary is midnight UTC (i.e., integer division by 86 400 000 ms). A ±1 day tolerance would allow "14 days ago" or "16 days ago" to pass, which the REQ prohibits. | Remove the `±1 day tolerance`. The assertion must be: inject `FakeClock` set to exactly 15 × 86 400 000 ms after the cache timestamp; assert the displayed string equals the string resource `R.string.location_cache_age_label` formatted with `15`. |
| F-14 | Low | §2.4 — GPS, step 1 | **"GPS fix available within the last 60 seconds" is an additional freshness constraint not present in the approved REQ.** REQ v0.2 §5.1 defines "current GPS fix" as "a fix received after the app opened in the current session" (no 60-second window). The FSPEC introduces a 60-second freshness cutoff which, if implemented, would cause True North to fall back to cached location immediately upon session resumption if the last fix was 61 seconds ago even though the fix was received in the same session. | Either (a) remove the "within the last 60 seconds" qualifier and replace with the REQ definition ("received after the app opened in the current session via `requestLocationUpdates`"), or (b) raise a question to PM/REQ to decide whether the 60-second window is intentional and have the REQ amended if so. |
| F-15 | Low | §3 — Business Rules / BR-10 | **`interference_flag` derivation in BR-10 conflicts with REQ v0.2 Scenario E and §5.3.1.** BR-10 states the flag is set if "field deviation ≥15% OR inclination deviation ≥3° at capture time." The REQ states the flag is set if "`InterferenceState` was `MODERATE` or `WARNING` at capture time." These are functionally equivalent only when there is no hysteresis: the `InterferenceDetector` has a 3-second clearance hysteresis, so it is possible for field deviation to drop below 15% while `InterferenceState` remains `MODERATE` during the clearance window. Under BR-10 the flag would be `false`; under the REQ it would be `true`. | Change BR-10 to reference `InterferenceState` directly: "`interference_flag` is `true` if `InterferenceState` is `MODERATE` or `WARNING` at the instant of the capture button tap." Remove the raw threshold comparison. This also aligns with the step 3b pre-capture warning trigger, which should likewise reference `InterferenceState` rather than raw deviation percentages. |
| F-16 | Low | §2.4 — GPS, location cache | **Location cache storage mechanism is not specified.** The FSPEC says the cache is stored in "encrypted local storage" but does not identify whether this is the existing encrypted Room database, `EncryptedSharedPreferences`, or a new storage sink. The existing `SettingsRepository` uses plain (non-encrypted) `SharedPreferences`. The Phase 1 `DatabaseKeyManager` uses `EncryptedSharedPreferences` only to store the Room cipher key. Engineering will independently choose a mechanism, which risks inconsistency. | Specify the cache storage mechanism in §2.4 or §6. Recommended: store the location cache in `EncryptedSharedPreferences` alongside the existing key store pattern (4 keys: lat, lon, alt, timestamp_ms), to avoid introducing a second Room table for a single-row cache. If the TSPEC is to make this decision, note it explicitly: "storage mechanism deferred to TSPEC §9." |
| F-17 | Low | §2.3 — Declination Info Panel | **"Using last known location (N days ago)" string in the main UI is not placed in the FSPEC.** The approved REQ v0.2 Scenario C requires the main UI to show `"Using last known location (N days ago)"` with string resource key `R.string.location_cache_age_label`. The FSPEC §2.3 places the cache age only in the declination info panel as `"N days ago"` (a different, shorter string), and the location state machine §5.3 shows no main-screen cached-location advisory. AT-C-02 only asserts the info panel content, not the main screen. The REQ-mandated main-screen string is unspecified. | Add a UI element to §2.3 or §5.1 (toggle state description) that describes where on the main compass screen the "Using last known location (N days ago)" string appears. Update AT-C to assert this string per REQ Scenario C. |
| F-18 | Low | §2.5 — Capture, step 5 and §6.1 | **`notes` field nullable constraint is mis-stated in §6.1 schema.** The schema table shows `Nullable = Yes` and `Constraint = "Length 0–500 or NULL"`, but then marks the field as `String?`. A `notes` field of empty string (`""`) and `null` are semantically different: the FSPEC step 6 says "empty by default" (implying `""`, not `null`). The schema should clarify whether an unfilled notes field is stored as an empty string or as `NULL`, as this affects Room entity definition and null-checks in Phase 4 display. | Decide and document: recommend storing `null` when the notes field is left blank (consistent with conditional fields pattern used for `lat`/`lon`). Update the constraint to `NULL or non-empty String (max 500 chars)` — i.e., an empty string should not be stored. |

## Questions

| ID | Question |
|----|---------|
| Q-01 | **`calibration_version` intent:** Was the FSPEC's redefinition of `calibration_version` as a WMM model name string intentional, or an inadvertent divergence from the REQ? If intentional, does the PM want to add a separate `wmm_model_version` field to the schema and keep `calibration_version` as the integer schema version per the REQ? |
| Q-02 | **`bearing_deg` precision:** `Float` gives approximately 4–5 significant decimal digits for an angle in [0, 360), which is ±0.001° precision. `Double` gives ±0.0000001°. Is `Double` necessary for WMM input accuracy, or is `Float` sufficient? The heading pipeline (`FusionResult.heading_deg`) already uses `Double` internally. |
| Q-03 | **`captured_at` storage format:** The REQ's "ISO 8601 UTC timestamp" implies a `TEXT` column storing a formatted string. The FSPEC correctly identifies `Long` (epoch ms) as the implementation-appropriate type (Room stores timestamps as `INTEGER`). Should the REQ be updated to reflect `Long/INTEGER` with ISO 8601 applied at presentation layer only? |
| Q-04 | **WMM warm cache definition:** The cold-start spec (REQ-NFR-08) distinguishes "warm cache" (WMM coefficients and GPS location already loaded) from "cold start." Does "already loaded" mean in-memory, or just on-disk? Loading 168 Gauss coefficients from assets and parsing them at cold start needs to be benchmarked. |

## Positive Observations

- The seven functional flows are internally consistent with each other — the state diagrams in §5 correctly reflect the decision trees in §2.
- Business rules table (§3) is well-formed: 20 discrete rules, each independently verifiable, with source references. This will materially reduce ambiguity during TSPEC authoring.
- The 35 acceptance tests map correctly to REQ scenarios A–F; AT-G extends coverage to edge cases (extreme latitude, WMM expiry, FakeClock boundary) that the REQ scenarios did not fully enumerate.
- BR-11 (single WMM2025 evaluation per location update, result cached for both declination and interference baseline) is an important performance constraint that correctly prevents double computation. This will be straightforward to verify in the TSPEC.
- The BearingRecord schema correctly includes `display_mode` as a Phase 3 forward-compatibility field, avoiding a migration later.
- The `fallbackToDestructiveMigration()` risk is correctly identified in REQ Risk P2-R4, and the FSPEC correctly calls for `Migration(1,2)`. Engineering should be aware that the **existing `LuopanDatabase.kt` currently uses `fallbackToDestructiveMigration()`** — this must be removed as part of Phase 2 work and will need a CI lint check per the REQ prohibition.

---

## Recommendation: Need Attention

> Six High or Medium findings are present. F-01 (`calibration_version` type conflict) and F-02 (permission rationale conflict) are direct contradictions with the approved REQ that will produce incorrect implementations if unresolved. F-03 (Phase 1 baseline mischaracterisation) will cause the FSPEC-DETECTUPGRADE flow to be implemented against a false description of the existing code, potentially introducing a regression in interference detection. F-05 (Clock/TimeSource duplication) and F-06 (MagneticFieldModel described as "existing") will cause unnecessary interface proliferation. All High and Medium findings must be resolved before the TSPEC is drafted.
