## Questions

| ID | Question |
|----|---------|
| Q-01 | Does bearing capture with Poor confidence **block** the save (requiring an explicit override tap) or merely **warn** before proceeding? Master REQ §8.2 says "capture blocked unless user overrides"; phase REQ §8 Scenario E says "a toast warns before saving." These are contradictory. The FSPEC must choose one and define the override UX explicitly. |
| Q-02 | Does the Phase 2 bearing record database share the Phase 1 calibration SQLCipher database (new table) or is it a separate encrypted database file? This has implications for the encryption key derivation strategy and the raw-bytes header test (PROP-PERSIST-02 pattern). |
| Q-03 | Is the extreme latitude confidence cap (§11.4, inclination ≥ 80°) implemented as a new sixth dimension in the `ConfidenceModel.compute()` call, or as a post-hoc override similar to the no-gyroscope cap? The Phase 1 confidence model has five dimensions + tilt + sensor state overrides. Adding a sixth dimension changes the PROPERTIES contract for all existing `PROP-CONF-*` tests. |
| Q-04 | For the `FakeClock` seam: should Phase 2 extend the existing `FakeTimeSource` used in Phase 1 interference detection tests, or introduce a separate `FakeClock` interface for calendar-based (date) operations vs. nanosecond-precision (timer) operations? The two use cases have different precision requirements. |
| Q-05 | When the WMM2025 model expires (post-2030) and the app falls back to `GeomagneticField`, does the north type toggle remain available with the "Android model" label, or is True North mode disabled entirely? The REQ says the label changes but does not say whether the feature remains active. |

## Positive Observations

- The Phase 2 REQ is well-structured and the six REQ §8 scenarios provide a clear skeleton for FSPEC authoring. Scenarios A through F map one-to-one to FSPEC sections the PM can draft immediately.
- The Phase 1 FSPEC is an excellent structural template: each FSPEC-* section has Actors, Preconditions, Behavioral flow, Business rules, Edge cases, Error scenarios, and Acceptance tests. The Phase 2 FSPEC should follow this structure identically to minimise review friction.
- The Phase 1 PROPERTIES document (76 properties) demonstrates mature test seam design: `FakeTimeSource`, `FakeSensorLayer`, protocol-based fakes throughout. Phase 2 only needs to extend this pattern — it does not need to invent a new one.
- REQ §9 Risk P2-R2 proactively flags the GPS permission denial path. This is exactly the kind of TE-visible risk that often gets missed at the REQ stage. The FSPEC author should treat it as a required flow, not an optional note.
- The Phase 1 encryption test strategy (PROP-PERSIST-02, raw-bytes header check) is a clean, non-rooted approach to verifying SQLCipher encryption that should be reused verbatim for the Phase 2 bearing record database.
