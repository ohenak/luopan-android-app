# Cross-Review: PM — TSPEC-luopan-p2-true-north-capture

**Reviewer role:** Product Manager
**Document reviewed:** TSPEC-luopan-p2-true-north-capture.md
**Date:** 2026-04-25

## Summary

The TSPEC provides solid coverage of all Phase 2 requirements and maps correctly to the FSPEC flows. The architecture choices are sound, the deferred-feature boundaries are respected, and the privacy and encryption requirements are implemented correctly. Three issues merit attention: a `captured_at` timestamp divergence from the FSPEC's stated semantics (FSPEC says "tap time" but the TSPEC acknowledges it records "save time"), a gap in the REQ traceability table for REQ-DECL-02 (the declination info panel's coordinate masking is implemented but not formally traced), and a minor ambiguity in how the `BearingCaptureUseCase` receives the `locationRepository` context needed to determine `includeLocation` at save time. None of these block approval, but the timestamp issue in particular should be resolved before implementation begins to avoid a Phase 4 history-screen surprise.

## Findings

| ID | Severity | Section | Finding | Recommendation |
|----|----------|---------|---------|----------------|
| PM-T-01 | Medium | §3.6 — `BearingCaptureUseCase`, note on `captured_at` | The FSPEC (§6.1 and BR-09) states: "The bearing snapshot is taken at the instant of the capture button tap." REQ §8 Scenario E says the record has `captured_at` = timestamp. The TSPEC acknowledges the divergence: "FSPEC says 'timestamp of capture button tap' but in practice records the save time." The gap between tap and Save can be several seconds (user types a name). In Phase 4, the history screen will display this timestamp as the capture time — users who tap at 10:00:00 and save at 10:00:14 will see the record labeled 10:00:14. This is a product fidelity issue. | Resolve the ambiguity now. The recommended fix is to add `tapTimestampMs: Long` to `BearingSnapshot` (populated in `CompassViewModel.captureBearing()` via `clock.nowMs()` at the instant the capture button is tapped) and use it in `BearingCaptureUseCase.execute()` for `captured_at`. The TSPEC already notes this option; commit to it rather than leaving it deferred. |
| PM-T-02 | Low | §0 — Requirements Traceability table | REQ-DECL-02 (Declination info panel — P1 requirement) is not listed as a traced requirement in any row of the §0 traceability table. The declination info panel content, coordinate masking (FSPEC §2.3 / BR-16), and "last updated" field are implemented across §3.7 (`CompassViewModel` `wmmModelId` / `locationState` / `DeclinationInfo`) and §7.2 (`DeclinationInfoBottomSheet`). The panel is built but its REQ-ID is absent from the traceability matrix. | Add a row to §0: `§7.2 — Declination Info Panel | REQ-DECL-02 | FSPEC-DECLPANEL §2.3`. This is a documentation gap, not an implementation gap. |
| PM-T-03 | Low | §3.6 — `BearingCaptureUseCase` constructor vs. `includeLocation` logic | `BearingCaptureUseCase` constructor takes `bearingRepository`, `modelProvider`, and `clock`. The `includeLocation` flag comes from `BearingSnapshot`. The snapshot is built in the UI layer, which can set `lat`/`lon` to null based on the GPS toggle. This is fine. However, the TSPEC does not explicitly state where the `lat`/`lon` fields of `BearingSnapshot` are populated (from `LocationRepository.resolvedLocation()` or from some other source). It is implied via `BearingCaptureUseCase` receiving `locationRepository` as a dependency listed in §1.3 dependency graph but NOT in the constructor of `BearingCaptureUseCase` itself (§3.6 constructor has only `bearingRepository`, `modelProvider`, `clock`). | Either (a) confirm that `lat`/`lon` are always passed in the `BearingSnapshot` from the ViewModel layer (removing the need for `LocationRepository` inside `BearingCaptureUseCase`) and update §1.3 accordingly, or (b) add `LocationRepository` to the `BearingCaptureUseCase` constructor. The §1.3 dependency graph currently shows `BearingCaptureUseCase → LocationRepository` which conflicts with the §3.6 constructor. Fix the inconsistency. |
| PM-T-04 | Low | §5.2 — WMM coefficient loading on main thread vs. cold-start budget | The TSPEC states that `Wmm2025Model` is constructed on `Dispatchers.IO` via `viewModelScope.async(Dispatchers.IO)` in `CompassViewModel.init {}`. However, §5.2 also states: "Loading in the constructor (called at app startup by `MagneticFieldModelProvider`) amortizes the cost before user interaction." These two statements slightly conflict — it is unclear whether `MagneticFieldModelProvider` itself is constructed eagerly on the main thread (which would trigger the constructor and the file I/O) or lazily. For REQ-NFR-08 (cold start ≤5 s), if the WMM parse happens synchronously on the main thread during `onCreate`, it could add 2–5 ms which is acceptable — but if the entire WMM init is deferred to a background `Deferred`, the first True North toggle tap must await it. Clarify the init path. | Add a sentence to §5.2 or §8.3 confirming the exact init sequence: does `MagneticFieldModelProvider` initialization happen in `CompassViewModel.init {}` on `Dispatchers.IO`, and does the `setNorthType(TRUE)` path `await` the deferred? This affects UX on first True North toggle tap on a cold start. |
| PM-T-05 | Low | §3.5 — `BearingRepository.getAll()` and `delete()` — Phase 4 features in Phase 2 | `BearingRepository` exposes `getAll()` and `delete()`, which are Phase 4 (history screen) features. REQ §6 defers REQ-CAPTURE-03 (history screen) to Phase 4. The TSPEC note says "present now to avoid a future migration" — this is a reasonable pre-stubbing approach, but it should be explicitly called out as a deferred-feature stub to ensure QA does not test these paths or product does not expose them prematurely. | Add a comment or annotation to `getAll()` and `delete()` in §3.5 marking them as "Phase 4 — stub only; not called from any Phase 2 UI path." Also confirm neither method is wired to any Phase 2 UI component. This ensures the deferred list (REQ §6 REQ-CAPTURE-03) is truly deferred. |
| PM-T-06 | Info | §6.3 — NETWORK / PASSIVE provider fallback not in FSPEC | FSPEC §2.4 step 1 specifies only `GPS_PROVIDER` and the 30-day cache. TSPEC §6.3 introduces a `NETWORK_PROVIDER` and `PASSIVE_PROVIDER` fallback for indoor scenarios. This is an engineering addition not described in the FSPEC or REQ. While the intent is user-friendly (better indoor coverage), it changes the location resolution semantics: a stale network fix (< 60 s old) would produce `LocationState.GpsFresh`, masking the fact that it may have much higher position uncertainty than a GPS fix. The 60-second "freshness" rule applied to a network fix could cause a significantly less accurate declination to be displayed as "GPS fix" in the info panel. | Either (a) limit the NETWORK/PASSIVE fallback to produce `LocationState.GpsCached` only (never `GpsFresh`) so the info panel correctly labels it as "Cached location", or (b) surface the provider type in `CachedLocation` so the panel can display "Network fix" vs "GPS fix". Confirm the FSPEC §2.3 source label table needs a new entry or clarify that "GPS fix" is the label for any system-provided location. This should be aligned with the PM/Product team before implementation. |
| PM-T-07 | Info | §3.7 — `CompassUiState` additions not fully specified | §3.7 lists new fields added to `CompassUiState` as comments (`north_type`, `declination_deg`, etc.) but does not provide the updated `CompassUiState` data class definition. This is acceptable for a TSPEC (the FSPEC owns the behavior, the TSPEC owns the implementation), but the omission means the reviewer cannot verify that all FSPEC-specified state (e.g., `wmmModelId`, `captureButtonEnabled`) is surfaced. | Consider adding a code block showing the full updated `CompassUiState` data class in §3.7 or a new §3.9. This will save implementation time by removing ambiguity and prevent back-and-forth during code review. |

---

## Recommendation: Approved with Minor Issues

The TSPEC is technically sound and well-aligned with REQ and FSPEC. PM-T-01 (captured_at tap-time vs. save-time) should be resolved in the TSPEC before the PLAN is authored. PM-T-03 (LocationRepository dependency graph inconsistency) should be corrected to prevent a confusing implementation hand-off. All other findings are documentation clarifications that can be addressed in the next TSPEC revision without blocking work.
