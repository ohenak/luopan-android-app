## Summary

Phase 3 introduces a custom Canvas-rendered luopan dial with six concentric rings, a mode-switcher architecture, pinch-to-zoom, long-press ring-visibility menus, 坐向 lock state, and two localization toggles — all on top of the existing single-Activity/single-ViewModel architecture that currently has no concept of display modes or a `LuopanView`. The requirements are directionally sound and the ring data tables are well-specified, but several high-priority implementation gaps would block engineering: the mode-switcher contract is entirely unspecified for P3, the `CompassUiState` has no fields for any luopan-specific state, `SettingsRepository` has no keys for the new toggles, the ring-visibility persistence decision is left open, and the `坐向 lock` confidence threshold in the phase REQ contradicts the master REQ. Medium-priority gaps cover thread-safety for the pinch-to-zoom scale factor, `onDraw` performance budget for six rings on the UI thread, and missing error/empty states for the mode-switcher. Low-priority gaps cover font asset loading and a minor acceptance-test inconsistency.

## Findings

### High Priority

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-01 | High | **No mode-switcher architecture defined.** The master REQ (REQ-DISPLAY-01) requires a mode switcher (tab bar or swipe), but the existing codebase has a single `CompassActivity` with `CompassRoseView` and no concept of a Luopan fragment, back-stack entry, or navigation graph. The phase REQ says "Accessible via mode switcher" (§2) but provides zero architecture guidance: no Fragment vs. Activity decision, no ViewPager/NavHost spec, no contract for how the Luopan mode receives heading updates from the shared `CompassViewModel`. Without this, engineering cannot start the skeleton without invention. | §2 Capability table; REQ-DISPLAY-04 |
| F-02 | High | **`CompassUiState` has no luopan-specific fields.** The readout panel (REQ-DISPLAY-05, §2) must expose 山 character + pinyin, 十二地支 sector + pinyin, 後天八卦 方位 (trigram + direction), 分金 (at High confidence), and confidence badge — all derived from the current heading. None of these fields exist in `CompassUiState`. The mapping logic (heading → sector lookup → character) is also unspecified: should it live in the ViewModel, a domain object, or the View itself? Without this contract, the ViewModel, the state data class, and the View cannot be authored independently. | §2 Numerical readout; REQ-DISPLAY-05 |
| F-03 | High | **坐向 lock confidence threshold contradicts master REQ.** Phase REQ §8 (Scenario E) states the lock is disabled only at Poor confidence (`Given confidence is "Poor", When user attempts to tap "Lock 向", Then: Button is disabled`), implying it is enabled at Moderate. Master REQ REQ-DISPLAY-06 states: "Locking is only permitted when confidence is 'High' or 'Moderate'; the button is disabled when confidence is 'Poor'." However, Phase REQ §5.1 (REQ-DISPLAY-06 row) marks this requirement as `P0 (elevated)` and the Key Spec says "disabled at Poor confidence" — consistent with master REQ. But Scenario E step 1 reads `Given confidence is "High"`, implying only High enables the lock. These three statements are internally consistent only if Moderate also enables the lock, but Scenario E never tests the Moderate path. Engineering needs an explicit acceptance criterion for Moderate confidence + Lock 向. | §5.1 REQ-DISPLAY-06; §8 Scenario E |
| F-04 | High | **`SettingsRepository` has no keys for the two new localization toggles.** REQ-L10N-02 (§5.4) requires "Show romanization" and "Show in my language" toggles. The existing `SettingsRepository` has keys only for `declination_mode`, `manual_declination_deg`, `display_format`, and `wake_lock_enabled`. No storage contract is defined for the new boolean flags, their default values, or their SharedPreferences key names. Without this, two independently developed components (the settings UI and the LuopanView) cannot agree on state. | §5.4 REQ-L10N-02; §10 Scenario F |
| F-05 | High | **Ring-visibility toggle persistence is explicitly deferred to engineering but not constrained.** Scenario G (§10) states: "hiding is not persisted across app restarts (or is persisted — engineering decision, but must be documented in settings)." This is a product decision disguised as an engineering decision. If ring visibility is persisted, it must be keyed in `SettingsRepository` and exposed in a settings screen. If it is session-only, the ViewModel must hold it in memory. Either path has significant UX implications: a practitioner who hides Ring 6 (六十分金) and restarts should not be silently returned to the cluttered six-ring view without expectation-setting. PM must decide; engineering cannot safely choose. | §10 Scenario G |
| F-06 | High | **No architecture spec for the `LuopanView` Canvas rendering thread model.** Six concentric rings with per-sector label layout, CJK font rasterization, and a continuously rotating dial represent a significant `onDraw` workload. The current `CompassRoseView` does all drawing on the UI thread at 20 Hz (per REQ-NFR-02). At six rings × 60 labels for Ring 6, `Canvas.drawText()` will be called 60+ times per frame with Noto Serif CJK TC (a large font requiring typeface loading). There is no spec for whether drawing is hardware-accelerated, whether the ring geometry should be pre-computed into `Path` objects in `onSizeChanged`, or whether a `SurfaceView`/`GLSurfaceView` should be used. If the naïve implementation drops below 20 Hz on API-26 mid-range hardware, the phase will fail REQ-NFR-02 with no mitigation path stated. | §6 Visual Specification; REQ-NFR-02; §10.3 |

### Medium Priority

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-07 | Medium | **Pinch-to-zoom scale factor is not threadsafe and has no persistence contract.** The interaction spec (§7) requires pinch-to-zoom 0.8×–2.0×. The scale factor must be written from a `ScaleGestureDetector` callback (UI thread) and read in `onDraw` (also UI thread, safe in isolation), but if a `SurfaceView` is used for performance the read/write will cross threads. More critically, whether the zoom level survives a configuration change (rotation) is unspecified — it must survive (user expectation), which requires either ViewModel state or `onSaveInstanceState`. Neither is mentioned. | §7 Interaction Behavior |
| F-08 | Medium | **Long-press ring visibility menu UX is completely unspecified beyond "opens a menu."** The spec says "Long-press on the dial opens a menu to hide/show individual rings" (§7). No details: Is this a `PopupMenu`, a `BottomSheetDialog`, a custom overlay? How are the six rings labelled in the menu (by Chinese name, English name, ring number)? What happens on devices that don't register long-press on a custom View by default (accessibility mode)? An accessibility alternative must be considered for TalkBack users. | §7 Interaction Behavior |
| F-09 | Medium | **Mode-switcher transition error and loading states are not specified.** REQ-DISPLAY-01 says mode transition must complete in <300 ms. The phase REQ does not specify: (a) what to show while the Luopan View is inflated and its first `onDraw` is pending (blank screen? progress spinner?), (b) what happens if the user switches mode during a configuration change, and (c) whether sensor data continues uninterrupted during mode switches (it should, but the heading flow from `CompassViewModel` to the new Luopan view must be wired up before the 300 ms budget expires). | §2; REQ-DISPLAY-04 |
| F-10 | Medium | **`分金` label content is never defined in Phase 3 scope.** REQ-DISPLAY-05 (§5.1) says "分金 shown only at High confidence" and REQ-LUOPAN table (§5.3) lists 六十分金 as Ring 6 with 60 divisions at 6°. But the actual 60 label strings are not included in the phase REQ or referenced master REQ section. The master REQ §10.2c gives the 二十四山 table (Ring 5) but no 六十分金 (Ring 6) label table. OQ-P3-01 (§11) flags this as an open question pending feng shui consultant review. Engineering cannot hard-code 60 labels without the authoritative table. This is a content dependency, not an engineering risk per se, but it blocks implementation of both Ring 6 and the numerical readout panel 分金 field. | §5.3; OQ-P3-01; REQ-DISPLAY-05 |
| F-11 | Medium | **NFR coverage for Luopan Mode is absent.** The master REQ NFRs (§7) apply to "Active compass display (all sensors running, screen on, Modern Mode)" for battery (REQ-NFR-03) and cold-start (REQ-NFR-08). Luopan Mode with six rings and continuous rotation will consume more GPU/CPU than Modern Mode. Phase 3 adds no NFR targets specific to Luopan Mode — no frame-drop budget, no additional battery allowance, no `onDraw` time budget. Without Luopan-specific NFR numbers, QA cannot accept or reject the implementation on performance grounds. | §7 (master); REQ-NFR-02/03 |
| F-12 | Medium | **Gold tick mark for locked 向 on the rotating dial requires coordinate-space math not specified.** Scenario E (§8) requires "a gold tick mark appears on the dial at the locked 向 position." The dial rotates continuously; the tick mark must remain visually anchored to a specific bearing on the dial (not the screen), rotating with the dial. This requires drawing in the dial's rotated coordinate space — not as a fixed overlay. The spec does not acknowledge this implementation complexity. If the tick mark is naively drawn at a fixed screen position it will appear stationary while the dial rotates, which is incorrect. | §8 Scenario E; §10.5 |

### Low Priority

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-13 | Low | **Font asset loading is unspecified.** §6 specifies "Noto Serif CJK TC (or equivalent)" at specific sizes. Noto Serif CJK TC is not bundled by default in Android's system fonts on all API 26+ devices (it was added to AOSP later, and OEM ROMs vary). If the font is not present, `Typeface.create("Noto Serif CJK TC", ...)` silently falls back to the system default, which may not render Traditional Chinese characters correctly. The spec should state whether the font must be bundled as an app asset (adding ~10 MB to APK) or whether a system font lookup with graceful fallback is acceptable. | §6 Visual Specification; §10.3 |
| F-14 | Low | **Acceptance test Scenario B numerical readout format is inconsistent with REQ-DISPLAY-05.** Scenario B (§10) expects `"☲ 離 · 南 · 午 (Wǔ) · 180.0° · True N · [confidence]"` — but REQ-DISPLAY-05 lists 山 (char + pinyin) and 十二地支 sector (char + pinyin) as separate readout fields, not concatenated in this format. "午" is a 山 in the 二十四山 ring (Ring 5, center 180°) AND a 十二地支 character (Ring 4, center 180°). The Scenario B string conflates these two rings. Engineering needs clarity on whether both 山 (二十四山 ring) and 時辰 (十二地支 ring) are shown at 180° and how the readout panel distinguishes them. | §10 Scenario B; REQ-DISPLAY-05 |
| F-15 | Low | **Scenario D uses undefined term "all conditions met per master REQ §8.1."** Scenario D (§10) says `Given confidence is "High" (all conditions met per master REQ §8.1)`. This is correct by reference but difficult to reproduce in an instrumented acceptance test without a test harness that can inject a `ConfidenceModel` returning `HIGH`. The acceptance criterion is not autonomously testable — it requires either a real device in a controlled magnetic environment or a dependency-injection hook not yet specified in the architecture. | §10 Scenario D |

## Positive Observations

- The six-ring data tables (§5.3, §10.2a, §10.2b, §10.2c) are fully populated with sector boundaries, center bearings, characters, and pinyin. Engineering can hard-code these as compile-time arrays without ambiguity.
- The pointer-fixed/dial-rotates paradigm is stated unambiguously and consistent with physical luopan operation. The counter-clockwise-on-clockwise-rotation acceptance criterion in Scenario A is testable.
- Risk P3-R2 (§11) correctly identifies the LUT-vs-computation risk for sector assignment and mandates the safer implementation. This is good engineering guidance embedded in the REQ.
- The 子 sector wrap-around (345°–15°) edge case is called out explicitly in both the data table and a prose note. This is the most common source of off-by-one bugs in circular sector lookups.
- The visual styling table (§6) provides hex colors and sp sizes that are directly usable in XML and Canvas paint setup with no interpretation required.
- Deferring 六十龍 / 七十二龍 (§9) pending hardware accuracy confirmation is the correct call and removes a high-risk scope item from P3.

## Recommendation

Need Attention

Six High findings must be resolved before engineering can begin authoring the TSPEC. F-01 (mode-switcher architecture), F-02 (UiState contract), F-03 (confidence threshold contradiction), F-04 (settings keys), F-05 (ring-visibility persistence), and F-06 (Canvas rendering thread model) collectively represent the skeleton of Phase 3 implementation. Without decisions on these items, parallel authoring of the TSPEC, PLAN, and PROPERTIES will produce artifacts that contradict each other at integration time.
