## Findings

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-01 | High | **`FakeDriftDetector` subclasses concrete `DriftDetector`, not `IDriftDetector`.** TSPEC §9.3b shows `FakeDriftDetector` as `class FakeDriftDetector(…) : DriftDetector(FakeClock(0L))`, i.e., subclassing the concrete class. PROP-CAL-037 requires `CompassViewModel` to accept `IDriftDetector` so that a fake can be injected. But no `IDriftDetector` interface is defined anywhere in the TSPEC — `DriftDetector` is a concrete class with no declared interface. PROP-CAL-036 covers the enum and `onFrame()` signature but defines the test file as `IDriftDetectorContractTest.kt`, which implies the interface should exist. Without an `IDriftDetector` interface in the TSPEC, PROP-CAL-037 is unimplementable as written: `CompassViewModel` cannot accept an interface type that does not exist. The PROPERTIES document promotes this to a mandatory contract but the architectural artifact (the interface itself) is absent from the TSPEC. This is a direct blocker for PLAN task B-2a. | PROP-CAL-036, PROP-CAL-037; TSPEC §5.3, §9.3b |
| F-02 | High | **`BearingHistoryViewModel` receives `BearingDao` directly, bypassing the repository layer.** TSPEC §6.3 constructs `BearingHistoryViewModel(dao: BearingDao)` with a raw DAO. PROP-HIST-001, PROP-HIST-004, PROP-HIST-031–033 all specify DAO method calls as the pass condition. However, the existing codebase uses a `BearingRepository` interface (`BearingRepositoryImpl` wraps `BearingDao`). Injecting the DAO directly into the ViewModel breaks the existing layered architecture and makes it impossible to stub at the repository boundary in tests without touching the DAO. The `FakeBearingDao` (TSPEC §9.2) implements `BearingDao` — a Room interface — which requires the DAO to be an interface rather than an annotated abstract class (Room generates the implementation). If `BearingDao` is a Room `@Dao` abstract class, `FakeBearingDao` cannot subclass it and also implement it cleanly as a fake. The PROPERTIES debounce unit tests (PROP-HIST-031, PROP-HIST-032) assert against `searchFlowCalls.size` on `FakeBearingDao`, but if the ViewModel takes a `BearingRepository`, these tests cannot observe the DAO directly. The architecture must be clarified: either (a) `BearingHistoryViewModel` takes a `BearingRepository` interface (which `FakeBearingRepository` can implement) and PROP-HIST-031–033 are restated in terms of repository calls, or (b) the DAO is promoted to an interface and the existing `BearingRepositoryImpl` wrapping is removed for this use site. Neither option is documented; both require PROPERTIES changes. | PROP-HIST-031, PROP-HIST-032, PROP-HIST-033, PROP-HIST-047–049; TSPEC §6.3, §9.2 |
| F-03 | High | **PROP-CAL-040 claims persistence survives process death using a unit test (`CompassViewModelDriftTest.kt`), which cannot verify SharedPreferences persistence across process boundaries.** The pass condition states: "After dismiss, process kill, and ViewModel recreation: `settings.driftCooldownTimestampMs > 0L`." Unit tests using `FakeSettingsRepository` cannot simulate process death — they run in-process and the fake is a simple in-memory object. A `FakeSettingsRepository` reset between test calls is not equivalent to a process kill followed by SharedPreferences read from disk. This property should be reclassified as an Integration test using a real `SettingsRepository` backed by an instrumented `SharedPreferences` context (or Robolectric with real SharedPreferences), not a JVM unit test. As currently specified, the pass condition is structurally impossible to satisfy at the stated Unit level. | PROP-CAL-040; TSPEC §4.7 |
| F-04 | Medium | **PROP-HIST-022 asserts no `notifyDataSetChanged()` call via source inspection ("no `notifyDataSetChanged()` call in adapter source"), but the stated test file is `BearingAdapterFormatTest.kt` (a runtime unit test).** Source-level assertions cannot be made at runtime. The property conflates two different verification techniques: a runtime assertion (spy/call count) and a static source check. The pass condition says "Spy on adapter: `notifyDataSetChanged` call count == 0" — but also says "no `notifyDataSetChanged()` call in adapter source." These are different mechanisms. The runtime spy approach is correct and sufficient. The source-check phrasing should be removed from the pass condition to avoid implementer confusion about what the test must assert. Additionally, `ListAdapter` is an Android framework class; subclassing it in a JVM unit test (without Robolectric) requires either Robolectric or a manual stub of `ListAdapter`, which is non-trivial. The test level may need to be Integration rather than Unit. | PROP-HIST-022; TSPEC §7.2 |
| F-05 | Medium | **PROP-HIST-071 specifies a macrobenchmark pass condition of "p99 ≤ 16 ms" but the TSPEC specifies no macrobenchmark module or baseline profile setup.** The TSPEC §2 lists `BearingHistoryBenchmark.kt` in the new files table but provides no macrobenchmark Gradle module configuration (`:macrobenchmark` module, `build.gradle.kts` additions, `BaselineProfilePlugin`, `CompilationMode`). Macrobenchmark requires a separate `:macrobenchmark` Gradle module with `testInstrumentationRunner = "androidx.benchmark.junit4.AndroidBenchmarkRunner"` and a device/emulator with profiling enabled. Without this infrastructure, `BearingHistoryBenchmark.kt` cannot compile or run. The PLAN references a benchmark build type was already added (see recent commit `fix(ci): add benchmark build type to :app for macrobenchmark workflow`), but the PROPERTIES pass condition cites `FrameTimingMetric` which is a Macrobenchmark API — not available in standard instrumented tests. If the benchmark module already exists from prior CI work, the PROPERTIES document should reference it explicitly and the TSPEC should confirm the module is in place. If it does not exist, this is a blocker for PROP-HIST-071. | PROP-HIST-071; TSPEC §2 |
| F-06 | Medium | **PROP-HIST-064 tests `updateListState()` at Unit level in `BearingHistoryViewModelTest.kt`, but `updateListState()` is specified in the TSPEC as a private Fragment method (§7.1), not a ViewModel method.** Unit testing a private Fragment method in a JVM ViewModel test is impossible. The function is defined in `BearingHistoryFragment.kt` as a private UI helper that directly references view objects (`searchBar`, `recyclerView`, `emptyNoRecords`, `emptyNoResults`). It cannot be called from `BearingHistoryViewModelTest.kt` without a running Fragment context. The property must either (a) be reclassified as E2E (instrumented) with the Fragment test seeding the correct state, or (b) the `updateListState()` logic must be extracted into the ViewModel as a pure function producing a visibility descriptor data class (with no View references), making it JVM-testable. As specified, the pass condition is unreachable at the Unit level. | PROP-HIST-064; TSPEC §7.1 |
| F-07 | Medium | **PROP-SENSOR-037 claims `buildJsonFromSensors()` is a "pure function (no `Context` or `SensorManager` dependency)" and testable as a JVM unit test, but its signature in TSPEC §5.5 takes `List<Sensor>` where `Sensor` is `android.hardware.Sensor` — a final Android framework class with no public constructor.** A JVM unit test cannot instantiate `android.hardware.Sensor` objects. The TSPEC comment says "Tests inject a list of `FakeSensorDescriptor` objects" — but the function signature takes `List<Sensor>`, not `List<FakeSensorDescriptor>`. Without Robolectric, there is no way to construct a `Sensor` instance in a JVM test. The TSPEC does not define `FakeSensorDescriptor` anywhere, nor does it show how `buildJsonFromSensors()` would accept it (since the signature is `List<Sensor>`, not a generic interface). Either: (a) the function must be refactored to accept `List<SensorDescriptor>` where `SensorDescriptor` is a data class or interface the test can construct (requiring a TSPEC change), or (b) the test must be reclassified as Robolectric/Integration. The pass condition "callable directly with a `List<Sensor>` argument in a JVM test without robolectric or instrumentation" is structurally impossible with the Android SDK's `Sensor` class. | PROP-SENSOR-037; TSPEC §5.5 |
| F-08 | Medium | **PROP-HIST-042 requires Snackbar duration verification "via reflection or wrapper" — but Android's `Snackbar` class makes the internal duration field (`mDuration`) package-private and obfuscated in release builds.** Reflection-based access to `mDuration` is brittle: field name may change between material-components versions, will fail under strict reflection policies (Android 14+ restricts non-SDK interfaces), and cannot be performed without reflection exemptions or a custom Snackbar subclass. The TSPEC §7.3 specifies `setDuration(5000)` directly, which is testable by asserting the Snackbar disappears within a bounded time window (behavioral test), or by wrapping `Snackbar.make` behind a testable factory. The pass condition should be rewritten to use a behavioral assertion (Snackbar disappears between 4 and 6 seconds after show) or an injectable Snackbar factory, rather than reflection. As stated, the reflection approach is high-risk and likely to fail on Android 14+ devices. | PROP-HIST-042 |
| F-09 | Low | **PROP-HIST-003 pass condition references `RecyclerViewItemCountAssertion(10)` as a hardcoded count.** The property itself says "display the record with the maximum `captured_at` value at position 0 when the fragment is first launched" — this is the core assertion. The count of 10 is a seed data detail that belongs in the test fixture, not in the property pass condition. If the seed data changes from 10 records to 20, this property would need to be updated despite the behavioral invariant (newest at position 0) being unchanged. Decouple the count assertion from the behavioral assertion by removing the specific count from the pass condition or restating it as "RecyclerView contains at least 1 record." | PROP-HIST-003 |
| F-10 | Low | **PROP-CAL-001 pass condition hardcodes the banner text as "Your calibration is 31 days old — consider recalibrating," which duplicates the string template in the TSPEC.** If the string resource `banner_cal_age` changes wording (e.g., "recalibrate now" instead of "consider recalibrating"), PROP-CAL-001's pass condition is stale. The pass condition should reference the string resource key and assert the substituted value contains the correct day count, rather than a hardcoded English sentence. This is a maintainability risk, not a correctness issue for this iteration, but it will silently break the property on any copy change. | PROP-CAL-001; TSPEC §7.5 |

## Positive Observations

- **Cross-review findings from TE PLAN-v2 are correctly locked as properties.** PROP-CAL-016 (correct AT-CAL-03-B2 sequencing), PROP-CAL-036 (enum constants and onFrame signature), PROP-CAL-037 (IDriftDetector injection) are cleanly derived from the TE PLAN-v2 F-01/F-02/F-03 findings. The intent to prevent spec defects from propagating to implementation is sound.

- **The debounce properties (PROP-HIST-031, PROP-HIST-032, PROP-HIST-033) are among the most precisely specified in the document.** The pass conditions give exact timing scenarios (t=0, t=200, t=250, t=300, t=500, t=550 ms) that leave no implementation ambiguity. This level of specificity is appropriate for the 300 ms debounce invariant.

- **PROP-CAL-014 and PROP-CAL-035 together precisely pin the `AccelerometerVarianceTracker` formula.** Specifying both the combined-magnitude requirement (PROP-CAL-014) and the population-variance formula (PROP-CAL-035) with concrete numeric examples ([3, 5, 4] → 0.6667) eliminates the risk of a silent formula substitution that would shift the `ACCEL_VARIANCE_THRESHOLD` effectiveness.

- **PROP-CAL-016 correctly captures the "re-evaluate-each-frame" rule** from TSPEC §5.3, which was absent from the original FSPEC. The two-step pass condition (first `onFrame()` with deviation=10% stays COUNTING, then `onFrame()` with 11% triggers) precisely tests the boundary without relying on a single-call pass.

- **PROP-SENSOR-020 through PROP-SENSOR-024 form a coherent failure-retry contract.** The chain — IOException → key not updated → retry on next call — is fully covered with specific assertions at each step. PROP-SENSOR-021 (SecurityException handled) and PROP-SENSOR-022 (catch-all Exception block) add real safety margin beyond what the FSPEC specifies.

- **Coverage matrices (§5.1 and §5.2) are complete and navigable.** Every named FSPEC acceptance test maps to at least one property, and every property maps back to at least one REQ. The gaps in §6.2 are explicitly called out with justification, which is the correct practice.

- **Empty-state properties (PROP-HIST-060 through PROP-HIST-064) cover all four state combinations** with specific View visibility assignments, including the subtle detail that the search bar must be `GONE` (not `INVISIBLE`) in State A. The four-branch coverage in PROP-HIST-064 references TSPEC §7.1 directly.

## Recommendation

**Need Attention**

> **F-01 (High):** The `IDriftDetector` interface must be declared in the TSPEC before PROP-CAL-036 and PROP-CAL-037 can be implemented. The PROPERTIES document requires an interface that does not exist in the architecture. The TSPEC §5.3 and PLAN B-2a must be updated.
>
> **F-02 (High):** The `BearingHistoryViewModel` injection boundary (DAO vs. Repository) must be resolved before PROP-HIST-031–033 and PROP-HIST-047–049 can be meaningfully implemented. If the ViewModel takes a raw `BearingDao`, `BearingDao` must be declared as an interface (not an abstract class) so `FakeBearingDao` can implement it.
>
> **F-03 (High):** PROP-CAL-040 must be reclassified as Integration (instrumented) with a real `SettingsRepository` and real SharedPreferences. A JVM unit test with a fake settings object cannot verify process-death persistence.
>
> **F-04 through F-08 (Medium):** These must be resolved before the corresponding PLAN tasks execute. The most impactful are F-06 (`updateListState()` is unreachable from a ViewModel unit test) and F-07 (`buildJsonFromSensors()` cannot accept `List<Sensor>` in a JVM test). Both require either architecture changes or test level reclassification.
