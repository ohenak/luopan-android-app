## Findings

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-01 | High | `BearingDao` currently exposes only `suspend fun getAll(): List<BearingRecord>` — no `Flow` overload exists. The FSPEC mandates a `Flow<List<BearingRecord>>` DAO method for reactive updates (live insert, swipe-undo restore, empty-state transition). Engineers must add a new `@Query … fun getAllFlow(): Flow<List<BearingRecord>>` to `BearingDao` and update `LuopanDatabase` schema version. This is a required contract change not explicitly called out as a code-change task in the FSPEC. | FSPEC-HIST-01 Business Rules — "Reactive updates"; FSPEC-HIST-04 "Live transition" |
| F-02 | High | `CalibrationWizardActivity.saveCalibration()` calls `repo.save(result, System.currentTimeMillis())` and writes to `CalibrationRecord` without any `expected_field_ut` field. The FSPEC/REQ requires adding `expected_field_ut: Float` to `CalibrationRecord`, a new `sphereRadius_uT: Float` field to `CalibrationResult`, and `CalibrationRepository.toRecord()` to map it. None of these changes exist in the codebase. The FSPEC describes this as a "three-file change" but the write-path in `CalibrationWizardActivity` itself also needs updating (it must pass `sphereRadius_uT` that comes out of `fitEllipsoid`). Currently `fitEllipsoid` returns a `CalibrationResult` without `sphereRadius_uT`, so the sphere radius `r` is a local variable that is discarded. The FSPEC implicitly requires a four-file change: `CalibrationEngine` (expose `r` via new field in `CalibrationResult`), `CalibrationResult` (new field), `CalibrationRepository.toRecord()` (map it), and `LuopanDatabase` DB version bump with `MIGRATION_2_3`. Engineers who read only the FSPEC may miss the `CalibrationWizardActivity` write-path change because it is not explicitly enumerated. | FSPEC-CAL-03 Preconditions §3; REQ §5.3 "three-file change" note |
| F-03 | High | `CompassActivity.wireTabNavigation()` has hardcoded `when (tab.position)` with only `TAB_MODERN = 0` and `TAB_LUOPAN = 1`. Adding `BearingHistoryFragment` as tab 2 requires: (a) adding a third `TabItem` to `activity_compass.xml`, (b) adding `TAB_HISTORY = 2` to the `CompassActivity` companion object, (c) wiring the `when` branch to `navController.navigate(R.id.dest_history)`, and (d) handling `dest_history` in the `addOnDestinationChangedListener` block. The FSPEC specifies the navigation destination contract in FSPEC-NAV-01 but does not enumerate the mandatory `CompassActivity` code changes. Without all four, the new tab either does nothing or causes an unhandled-position silent no-op. | FSPEC-NAV-01 "Tab Layout"; activity_compass.xml (currently 2 TabItems) |
| F-04 | Medium | `loadCalibrationAge()` in `CompassViewModel` is declared `private`. The FSPEC states it must be promoted to `internal` so `BearingHistoryFragment` (same module) can call it. This is the correct fix, but the FSPEC says "No new ViewModel class is required for the banner and calibration-age state." Currently the only way to expose `calibrationAgeDays` to a UI element is through `CompassUiState.calibration_age_days`, which is already a field in the shared `StateFlow<CompassUiState>`. The FSPEC does not specify a separate `StateFlow` for the age banner; engineers will need to decide whether `BearingHistoryFragment` observes `uiState.calibration_age_days` (already available) or waits for a distinct `loadCalibrationAge()` call result. The boundary between `BearingHistoryFragment`'s banner trigger and the shared sensor-loop value is ambiguous. | FSPEC-NAV-01 "ViewModel Ownership"; FSPEC-CAL-01 Behavioral Flow §2 |
| F-05 | Medium | The FSPEC specifies that the drift banner appears "exclusively in `BearingHistoryFragment`" and `DriftDetector` runs "continuously in the background (injected into `CompassViewModel`)." However, `DriftDetector` consumes the accelerometer variance of the combined 3-axis magnitude from the sensor loop running in `CompassViewModel.startSensorCollection()` on `Dispatchers.Default`. The FSPEC does not specify how `DriftDetector.onFrame()` receives the variance value — whether it is computed inside `DriftDetector.onFrame(variance, magnitude, interferenceState, expectedFieldUt)` or whether `CompassViewModel` computes variance externally and passes it. The `NoiseVarianceTracker` already exists and tracks noise variance, but it tracks magnitude variance for confidence scoring purposes. The FSPEC must clarify whether `DriftDetector` accepts a pre-computed `accVariance` argument (computed by a dedicated stationary-variance tracker over the accelerometer magnitude) or owns its own accumulator. Using the existing `NoiseVarianceTracker` for this purpose would be semantically incorrect — it tracks magnetometer noise, not accelerometer stationary variance. | FSPEC-CAL-03 Preconditions §1 |
| F-06 | Medium | The FSPEC states the drift banner appears only in `BearingHistoryFragment`, but `DriftDetector` is injected into `CompassViewModel`, which is Activity-scoped and always running. No mechanism is specified for what happens to `DriftEvent.TRIGGERED` emissions when the user is NOT on the History tab. If the ViewModel emits `TRIGGERED` while the user is on the Modern tab and then navigates to History, will the banner appear? The FSPEC says "When `DriftDetector` emits `DriftEvent.TRIGGERED`" but does not say whether the ViewModel buffers this event or holds it as state. If it is a `SharedFlow` (no replay), the event is lost if no subscriber is active. If it is a `StateFlow` or buffered flow, it remains and the History tab will show the banner when it next becomes visible. Engineers cannot determine the correct flow type from the FSPEC alone. | FSPEC-CAL-02 Behavioral Flow §2; FSPEC-CAL-03 State Machine |
| F-07 | Medium | The FSPEC specifies that `Sensor.getReportingMode()` returns an integer constant and that it must be serialised to a human-readable string (e.g., `"CONTINUOUS"`, `"ON_CHANGE"`). The mapping from integer to string is not provided. `Sensor.REPORTING_MODE_CONTINUOUS = 0`, `REPORTING_MODE_ON_CHANGE = 1`, `REPORTING_MODE_ONE_SHOT = 2`, `REPORTING_MODE_SPECIAL_TRIGGER = 3`. If an unknown integer is returned (e.g., a future OEM-specific mode), the FSPEC does not specify what string to write. Engineers will write differing fallback strings (e.g., `"UNKNOWN"`, `"$intValue"`, or a crash) unless the expected mapping is made explicit. | FSPEC-SENSOR-01 File Schema; sensors[].reporting_mode |
| F-08 | Low | AT-HIST-02-B specifies that the DAO query is "invoked exactly once" after typing "n", "o", "r" at 100 ms intervals. The FSPEC says the debounce is applied client-side or via a new Room query. If the implementation uses SQL `LIKE` filtering rather than in-memory filtering, the assertion "DAO query is invoked exactly once" is the correct testable outcome. However, if the implementation filters client-side from an already-emitting `Flow<List<BearingRecord>>`, the DAO is never additionally called — the observable is already subscribed. Engineers need to decide the filtering strategy before writing the test; the FSPEC says "or via a new Room query" without committing to one approach. This creates a fork in the test design. | FSPEC-HIST-02 Behavioral Flow §3; AT-HIST-02-B |
| F-09 | Low | The FSPEC states the search bar is "hidden or disabled" in State A (zero records). This is a binary implementation choice with different UI implications: hiding it avoids a confusing affordance; disabling it leaves the UI structure intact for transitions. The FSPEC does not specify which approach is required, and there is no acceptance test that asserts the search bar's visibility/enabled state in State A. | FSPEC-HIST-04 Business Rules "Search bar in State A" |
| F-10 | Low | `BearingHistoryFragment` may own a `BearingHistoryViewModel` for list, search, and delete state. The FSPEC says this "may additionally" be the case. Engineers need to decide whether `BearingHistoryViewModel` is required or optional, and if required, whether it uses `viewModels()` (fragment-scoped) or `activityViewModels()`. The two choices have different lifecycle implications for search query persistence across navigation. FSPEC-HIST-02 says the query is "cleared" when the user navigates away and returns — but this is only true if the ViewModel is fragment-scoped. If Activity-scoped, the query persists. | FSPEC-NAV-01 "ViewModel Ownership"; FSPEC-HIST-02 Edge Cases "User navigates away" |
| F-11 | Low | AT-HIST-03-D ("Process death — no undo on restart") specifies an instrumented test that terminates the process while a Snackbar is visible. This is achievable via `ActivityScenario` + `UiDevice.pressHome()` + process kill, but the test setup is non-trivial and requires a `benchmark` or instrumented test with proper process kill APIs. The FSPEC does not note this test complexity. The behavior itself is correctly specified (undo in-memory only), but the test methodology should acknowledge the difficulty of reliably killing the process in an instrumented test without causing test infrastructure flakiness. | AT-HIST-03-D |

## Questions

| ID | Question |
|----|---------|
| Q-01 | Should `DriftDetector.onFrame()` accept a pre-computed accelerometer variance (computed by a new dedicated accumulator in `CompassViewModel`) or should `DriftDetector` own its own internal accumulator that receives raw accelerometer magnitude per sensor frame? The existing `NoiseVarianceTracker` is for magnetometer noise, not accelerometer stationarity — clarify which component owns the stationary-variance computation. |
| Q-02 | When `DriftDetector` emits `DriftEvent.TRIGGERED` while the user is NOT on the History tab, how is the event delivered to `BearingHistoryFragment` when it later becomes visible? Should the ViewModel expose this as a `StateFlow<Boolean>` (banner-visible flag, survives navigation) or a `SharedFlow` event (can be missed)? A state-based approach is almost certainly correct, but the FSPEC does not specify the flow type. |
| Q-03 | Is `BearingHistoryViewModel` a required separate class or is all list/search/delete state held in `CompassViewModel`? If a separate class is used, it should be `viewModels()` (fragment-scoped) to get the query-clear-on-navigate behavior specified in FSPEC-HIST-02. This should be made explicit rather than "may additionally." |
| Q-04 | For the `sensors[].reporting_mode` JSON field: what string should be written for an unrecognized integer value returned by `Sensor.getReportingMode()`? Suggested: `"UNKNOWN($intValue)"` to preserve the raw value for diagnostics. |
| Q-05 | Does `BearingHistoryFragment` observe `CompassUiState.calibration_age_days` (already in the shared `StateFlow`) to drive the age banner, or does it call `loadCalibrationAge()` explicitly and observe a separate result flow? If the former, the `internal` visibility change to `loadCalibrationAge()` is still needed for RESULT_OK refresh, but the initial display does not require an explicit call. This should be resolved before TSPEC authoring. |

## Positive Observations

- The FSPEC resolves all eight open questions from the SE v4 and TE v4 REQ cross-reviews cleanly and completely. The ViewModel ownership decision (`activityViewModels()` + `internal` visibility), the FakeClock precedent file (`CompassViewModelTest`), the dismiss-without-navigation affordance (X button), and the RESULT_OK "reset" semantics are all precisely specified.
- The `DriftDetector` state machine diagram (FSPEC-CAL-03) is unambiguous: all state transitions are enumerated, timer-reset behavior is explicit with "no hysteresis," and the `Clock` interface dependency is correctly identified with the production injection path (`CompassViewModel.Factory`).
- The `expected_field_ut > 0.0` guard for Condition B is a robust backwards-compatibility decision. Existing records (which will have `expected_field_ut = 0.0` after `MIGRATION_2_3`) are cleanly excluded without requiring a nullable column.
- The dual-banner simultaneous display rule (FSPEC-CAL-01/CAL-02) is precisely specified: two separate, non-merged UI elements; AT-CAL-01-G provides an explicit acceptance test for this case.
- Swipe-to-delete semantics are exactly specified (commit on swipe, single active undo, replacement-commit behavior for second swipe) with no ambiguity about the undo window or DB timing.
- The `SensorCapabilityLogger` version-gate pattern (`VERSION_CODE > storedVersion`) is consistent with the `SettingsRepository` pattern established in Phase 3 and is correctly specified to survive process death and data-clear scenarios.
- The inclination deviation display format (truncation toward zero, not floor, preserving sign for negatives) is explicitly specified with a worked negative example. This is a common gotcha and is handled correctly.

## Recommendation

**Need Attention**

Five findings at High or Medium severity require resolution before TSPEC authoring:

1. **F-01 (High):** The TSPEC must explicitly include adding `Flow<List<BearingRecord>>` to `BearingDao` as a code-change task. The plain `suspend getAll(): List<BearingRecord>` that exists today is insufficient.
2. **F-02 (High):** The "three-file change" enumeration in REQ §5.3 must be updated to acknowledge that `CalibrationWizardActivity` also requires changes (it must use `sphereRadius_uT` from `CalibrationResult` when calling `repo.save()`). The FSPEC should either correct the enumeration or explicitly state that `CalibrationWizardActivity` does not need changes and explain why.
3. **F-03 (High):** FSPEC-NAV-01 must enumerate the required `CompassActivity` changes: new `TAB_HISTORY = 2` constant, new `when` branch, new `dest_history` handling in `addOnDestinationChangedListener`, and new `TabItem` in `activity_compass.xml`. Without this, engineers may add the Fragment but not wire it to the tab.
4. **F-05 (Medium):** FSPEC-CAL-03 must specify `DriftDetector.onFrame()` signature and clarify who computes the accelerometer stationary variance. A dedicated `AccelerometerVarianceTracker` (analogous to the existing `NoiseVarianceTracker`) appears to be needed.
5. **F-06 (Medium):** FSPEC-CAL-02 must specify the `DriftEvent` delivery mechanism — `StateFlow<DriftBannerState>` (recommended) vs. `SharedFlow<DriftEvent>` — so engineers know whether the banner state is retained across navigation or ephemeral.
