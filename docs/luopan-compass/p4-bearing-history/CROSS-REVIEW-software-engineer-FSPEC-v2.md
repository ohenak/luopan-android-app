## Findings

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-01 | Medium | `loadCalibrationAge()` is declared `private` in `CompassViewModel`. The FSPEC specifies it must be promoted to `internal` (FSPEC-NAV-01 Business Rules, FSPEC-CAL-01 ViewModel State). Verified against the codebase: the method is `private fun loadCalibrationAge()` at line 423 of `CompassViewModel.kt`. The `internal` visibility change is correct for same-module access from `BearingHistoryFragment`. However, the FSPEC does not specify whether the method call from `BearingHistoryFragment` is safe to make on the main thread or must be dispatched — the existing implementation launches `Dispatchers.IO` inside a `viewModelScope.launch`, so calling it from a Fragment lifecycle callback (main thread) is safe. This is not explicitly stated; engineers implementing `BearingHistoryFragment` may add unnecessary dispatcher wrapping. | FSPEC-CAL-01 ViewModel State; FSPEC-NAV-01 Business Rules |
| F-02 | Medium | The drift detection timer is per-frame in `DriftDetector.onFrame()`, but the FSPEC does not specify the expected call cadence — whether `onFrame()` is called at the full sensor frame rate (up to 200 Hz, as `startSensorCollection()` processes every frame from `SensorLayer.frames()`) or at a downsampled rate. The `accVariance` threshold of `< 0.01 (m/s²)²` and the 60-second timer are both sensitive to call rate. The `AccelerometerVarianceTracker` is specified as a "rolling 5-second window (250 samples at 50 Hz)", which implies `onFrame()` will be called at the same 50 Hz accelerometer rate. However, `CompassViewModel.startSensorCollection()` collects from `SensorLayer.frames()` which currently delivers magnetometer + accelerometer fused frames — the actual rate is not specified as constant in the FSPEC. If the sensor pipeline delivers frames at 200 Hz instead of 50 Hz, the `AccelerometerVarianceTracker` window size of 250 samples would represent only 1.25 seconds, not 5 seconds, and the 60-second timer would tick faster than wall-clock seconds. The FSPEC must specify the expected frame rate at which `DriftDetector.onFrame()` is called, or make `AccelerometerVarianceTracker` window-duration-based (milliseconds) rather than sample-count-based. | FSPEC-CAL-03 DriftDetector.onFrame() Signature; FSPEC-CAL-03 Preconditions §1 |
| F-03 | Medium | The FSPEC specifies that `BearingHistoryFragment` registers its `ActivityResultLauncher` in `Fragment.onCreate()` (FSPEC-NAV-01 ActivityResultLauncher Registration). However, both the age-based and drift-based banners launch `CalibrationWizardActivity` from the same fragment. The FSPEC specifies a single `ActivityResultLauncher<Intent>` for both banners (FSPEC-NAV-01: "registers an `ActivityResultLauncher<Intent>`"). The `RESULT_OK` handler then branches: if from the age banner, call `loadCalibrationAge()`; if from the drift banner, call drift detector reset. The FSPEC does not specify how the fragment distinguishes which banner launched the wizard when `RESULT_OK` is received — both banners open the same `CalibrationWizardActivity`. The `RESULT_OK` handler must be able to distinguish the source, either by: (a) using two separate `ActivityResultLauncher` instances registered for each banner, or (b) including a flag in the launch `Intent` that is returned in the result `Intent`. The FSPEC is silent on this disambiguation mechanism. Without it, `RESULT_OK` will attempt to perform both side-effects (refresh age AND reset drift) regardless of which banner was tapped. | FSPEC-NAV-01 ActivityResultLauncher Registration; FSPEC-NAV-01 Navigation Flow |
| F-04 | Low | FSPEC-CAL-01 specifies that `calAgeBannerDismissed` is a `Boolean` field in `CompassViewModel` (not `BearingHistoryViewModel`). The FSPEC also states that `BearingHistoryFragment` is fragment-scoped for its own ViewModel (`viewModels()`). However, the dismiss logic for Condition A is: "Banner body tap → `RESULT_OK` → `loadCalibrationAge()` called → banner dismissed." The FSPEC does not specify what makes the banner `GONE` after RESULT_OK — it says `loadCalibrationAge()` is called, which updates `calibration_age_days`, but if the user recalibrated just now, the new age would be 0 days, which is ≤ 30, so the condition naturally evaluates to false. This path is self-consistent. But the FSPEC also says the `calAgeBannerDismissed` flag is set to `true` only on the Close/X button path, not on the RESULT_OK path. After RESULT_OK, the dismiss is achieved through `calibration_age_days` dropping to 0, not through the flag. This is correct but is never stated explicitly — the FSPEC implies it through the table rows in "Dismiss Mechanics" but does not say "after RESULT_OK, the banner is GONE because `calibration_age_days ≤ 30`." An engineer may add a redundant `calAgeBannerDismissed = true` on RESULT_OK, or conversely may fail to call `loadCalibrationAge()` thinking the flag alone is sufficient. | FSPEC-CAL-01 Dismiss Mechanics; FSPEC-CAL-01 Behavioral Flow §3 |
| F-05 | Low | AT-CAL-02-F tests that the drift banner does not appear on compass tabs by asserting `onView(withId(R.id.banner_drift_root)).check(doesNotExist())` from `ModernCompassFragment`'s perspective. This assertion will pass trivially if `ModernCompassFragment`'s layout has never had `banner_drift_root` in it — the view ID does not exist in the layout, so `doesNotExist()` is always true regardless of `driftBannerState`. The test as written does not actually verify that the drift banner *state* is being suppressed or not displayed on that tab; it only verifies a layout invariant. A more meaningful assertion would be to verify the banner view in `BearingHistoryFragment` is not visible while on the Modern tab (since the Fragment is in a stopped state). The test as specified is not wrong, but it may give false confidence. This is a test design issue in the FSPEC's acceptance test specification, not a behavioral ambiguity, but it should be noted for the TSPEC author. | AT-CAL-02-F; FSPEC-CAL-02 Business Rules "Banner host" |
| F-06 | Low | FSPEC-HIST-03 specifies that "The Snackbar auto-dismisses after exactly 5 seconds." The Snackbar duration is set via `Snackbar.LENGTH_LONG` (the standard constant) or a custom duration. `Snackbar.LENGTH_LONG` on Android is defined as 2750 ms, not 5000 ms. To achieve a 5-second dismissal, the implementation must use `setDuration(5000)` on the `Snackbar` builder. The FSPEC says "exactly 5 seconds" but does not specify that `LENGTH_LONG` must NOT be used (which would give 2.75 s). If an engineer uses `LENGTH_LONG`, AT-HIST-03-A (which checks that DB contains 4 records immediately after swipe, before Snackbar times out) would still pass, but AT-HIST-03-C (second swipe cancels first, checked before any timer elapses) would also pass — so neither test would catch a 2.75s Snackbar. The REQ says "5-second undo toast" (REQ-CAPTURE-03); the FSPEC should note that `setDuration(5000)` is required, not `LENGTH_LONG`. | FSPEC-HIST-03 Behavioral Flow §3; REQ-CAPTURE-03 |
| F-07 | Low | FSPEC-SENSOR-01 specifies `SensorCapabilityLogger` must expose a constructor-injectable `SensorFileWriter` interface for unit-testability (AT-SENSOR-01-E). The FSPEC specifies the interface signature as `interface SensorFileWriter { fun write(content: String) }`. However, the FSPEC also specifies the write path as writing to `Context.getFilesDir() / "sensor_profile.json"` (a file path operation). A `write(content: String)` interface that receives the full serialized JSON as a `String` is sufficient for testing the "failure does not update version key" behavior, but it abstracts away the file path. The FSPEC should confirm whether the `SensorFileWriter` interface is responsible for both choosing the file path and writing, or whether `SensorCapabilityLogger` computes the path and passes the content to the writer. If the writer owns the path, the interface needs a `Context` or path argument. If `SensorCapabilityLogger` owns the path, the interface as specified is sufficient. This is a minor design gap that will be resolved during TSPEC authoring but is worth flagging. | FSPEC-SENSOR-01 AT-SENSOR-01-E |

## Questions

## Positive Observations

- All seven v1 High and Medium findings are cleanly resolved in v0.2-draft. The resolutions are precise and implementable without PM involvement — a significant improvement from v0.1.
- The `DriftDetector` state machine (FSPEC-CAL-03) is now fully specified with an explicit function signature, clear precondition ownership (`AccelerometerVarianceTracker` vs `NoiseVarianceTracker` distinction is explicit and correct), and a complete state transition diagram with post-trigger IDLE reset behavior.
- The `StateFlow<DriftBannerState>` decision (FSPEC-CAL-02) is the correct architectural choice for a Fragment that may not be visible when the event fires. The initial value of `HIDDEN` and the explicit retention guarantee across tab switches are correctly specified.
- The dual-banner simultaneous display rule with an explicit acceptance test (AT-CAL-01-G) closes a behavioral corner case that many FSPECs leave ambiguous.
- The `SensorCapabilityLogger` version-gate and retry-on-failure contract are fully implementable from the FSPEC alone. The injectable `SensorFileWriter` interface requirement (AT-SENSOR-01-E) is a sound testability design decision.
- The `LIKE '%query%'` SQL strategy for search (FSPEC-HIST-02) is correctly specified with debounce behavior, case-insensitivity caveats for non-ASCII, and an explicit switch back to `getAllFlow()` on empty query. The `searchFlow(query)` DAO signature is given, removing any implementation ambiguity.
- The AT-HIST-03-D process-death test reclassification (manual with rationale + AT-HIST-03-E as automated approximation) is the right pragmatic balance between test coverage and test infrastructure reliability.
- The inclination deviation truncation-toward-zero behavior (not floor, with sign preservation) is specified with a negative worked example. The v1 positive observation stands — this remains a well-handled gotcha.

## Recommendation

**Approved with Minor Issues**

All v1 High and Medium findings are resolved. The three remaining Medium findings (F-01, F-02, F-03) require attention but are resolvable in the TSPEC without needing FSPEC revision:

- **F-01** (Medium — `loadCalibrationAge()` main-thread safety): Add a comment in the TSPEC noting the method is safe to call from Fragment lifecycle callbacks; no FSPEC change needed.
- **F-02** (Medium — `DriftDetector.onFrame()` call rate and window sizing): The TSPEC author must resolve the sensor frame rate assumption for `AccelerometerVarianceTracker` and make the window duration-based if the rate is not guaranteed at 50 Hz. Q-01 must be answered before the TSPEC is written.
- **F-03** (Medium — `ActivityResultLauncher` disambiguation between two banners): The TSPEC must specify whether one or two launchers are used and how source disambiguation is achieved. Q-02 must be answered before the TSPEC is written.

The Low findings (F-04 through F-07) are informational for the TSPEC author and do not block the FSPEC.
