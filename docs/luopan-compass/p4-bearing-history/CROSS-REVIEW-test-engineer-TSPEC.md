## Findings

| ID | Severity | Finding | Section ref |
|----|----------|---------|------------|
| F-01 | Medium | `AccelerometerVarianceTracker` uses population variance (`/ n`), not sample variance (`/ (n-1)`). The threshold in REQ-CAL-05 and FSPEC-CAL-03 is stated as `< 0.01 (m/s²)²` without specifying which estimator to use. With a 5-second window at ~50 Hz the difference is negligible (~0.4%), but the implementation choice must be documented as an explicit decision so the unit tests in `AccelerometerVarianceTrackerTest.kt` can assert the correct formula. Currently the TSPEC shows population variance but the test strategy section (§9.1) lists only "Window eviction, variance output, time-based" — no test name is provided for the variance-formula assertion. Without a named test for this, the formula can silently change. | §5.2, §9.1 |
| F-02 | Medium | The integration test `AT-CAL-INT-01` (§9.3) explicitly documents that `CompassViewModel.startSensorCollection()` wiring is NOT covered — the ViewModel calls `accVarianceTracker.update()` then `driftDetector.onFrame()` then checks the cooldown and sets `_driftBannerState`. This three-step chain (`TRIGGERED → cooldown-check → VISIBLE`) is stated as covered by "CompassViewModel unit tests separately," but no such test appears in the coverage matrix (§9.1) or anywhere in the test file assignments (§3.1). The gap is real and named: if the cooldown-check code in the sensor loop is broken, only a production bug would surface it. A unit test for `CompassViewModel` that injects a fake `DriftDetector` stub returning `DriftEvent.TRIGGERED` and asserts that `driftBannerState` emits `VISIBLE` (when cooldown is elapsed) is missing. | §9.1, §9.3 |
| F-03 | Medium | `BearingHistoryViewModel.deleteRecord()` and `undoDelete()` both use `viewModelScope.launch(Dispatchers.IO)`. The TSPEC acknowledges in §10 (Error Handling) that uncaught exceptions are "swallowed by the coroutine scope." No test covers the failure path: if `dao.delete()` or `dao.insert()` throws, the UI state (RecyclerView, Snackbar) may be inconsistent with the database. This is accepted for Phase 4, but the TSPEC should either: (a) name a test that asserts the silent-failure behavior (i.e., the Flow re-emits the prior state from Room, recovering the visible record), or (b) explicitly state that this path is out-of-scope for Phase 4 testing. As written, the error contract is documented in prose but has no corresponding test entry, making it invisible to implementers. | §6.3, §9.1, §10 |
| F-04 | Medium | The `AT-CAL-INT-01` test body (§9.3) uses `return@runTest` as an early success exit inside a `repeat` loop, then a trailing `fail()`. This pattern passes silently if `DriftEvent.TRIGGERED` is emitted on frame 1 due to a bug (e.g., a missing precondition guard), because the `return@runTest` fires immediately. A correct test should capture the first `TRIGGERED` event and also assert it was NOT emitted before the 60-second window elapsed — e.g., by confirming no event was returned during the first 59 seconds of frames. Without a pre-threshold negative assertion, this integration test cannot distinguish a correctly behaving detector from one that fires on every frame. | §9.3 |
| F-05 | Low | `BearingHistoryViewModel` (§6.3) tracks `pendingUndo: BearingRecord?` as an in-memory field. The coverage matrix (§9.1) assigns swipe/undo entirely to instrumented tests (`AT-HIST-03-A through E`). No unit test for `BearingHistoryViewModel` verifies the `pendingUndo` state transitions: `null → record` on `deleteRecord()`, `record → null` on `undoDelete()`, and `record → null` (without restore) on `commitDelete()`. These state transitions are pure ViewModel logic with no Android dependency and should be unit-tested without requiring a device. The instrumented tests test the integrated behavior; they do not isolate the ViewModel state machine. | §6.3, §9.1 |
| F-06 | Low | `DriftDetector.onFrame()` is specified to return `DriftEvent.RESET` when a precondition violation transitions the state from COUNTING → IDLE. The TSPEC states the ViewModel "may ignore" this event. No test asserts that the ViewModel correctly ignores it (i.e., that `driftBannerState` does NOT change when `RESET` is received while the banner is HIDDEN, and does NOT hide the banner when `RESET` is received while VISIBLE). Both are meaningful negative tests — the banner staying visible after a brief precondition violation is an explicit business rule (FSPEC-CAL-02: "auto-hide only on explicit dismiss or RESULT_OK"). | §5.3, §9.1 |
| F-07 | Low | The TSPEC assigns `AT-SENSOR-01-E` (write failure) to unit tests and `AT-SENSOR-01-A through D` to instrumented tests. However, `AT-SENSOR-01-E` uses `SensorCapabilityLogger(context, settings, fakeWriter)` in a unit test — but `SensorCapabilityLogger.buildJson()` calls `context.getSystemService(Context.SENSOR_SERVICE)`, which requires an Android context. This means the unit test as specified will fail on a JVM-only test runner. The test must either: (a) use Robolectric (consistent with `CalibrationRepositoryTest`), or (b) extract `buildJson()` behind an injectable interface so the context dependency is replaced in unit tests. The TSPEC does not specify which approach. Without resolving this, `AT-SENSOR-01-E through G` are not runnable as JVM unit tests. | §5.5, §9.1, §3.1 |
| F-08 | Low | The coverage matrix (§9.1) lists the Room migration v2→v3 as covered by `LuopanDatabaseMigrationTest` addition, but does not specify what exactly will be tested. The existing `LuopanDatabaseMigrationTest` tests v1→v2 (verifying that `bearing_records` is created and `calibration_records` survives). The v2→v3 migration test must verify: (a) `expected_field_ut` column is added, (b) existing calibration rows have `expected_field_ut = 0.0` after migration, and (c) `CalibrationRecord.expected_field_ut` round-trips correctly via the DAO. None of these assertions are named in the TSPEC, leaving the implementer to guess what the test must verify. | §9.1, §4.1 |
| F-09 | Low | The `FakeBearingDao` (§9.2) implements `searchFlow()` by filtering `_records` in memory using `String.contains()` with `ignoreCase = true`. This correctly matches the `LIKE '%query%'` SQL behavior for ASCII, but does not faithfully reproduce the Room `LIKE` operator's behavior for non-ASCII input (where the FSPEC says case-insensitivity is best-effort). This is an accepted limitation, but the fake's filtering logic will pass tests for non-ASCII queries that the real DAO might fail, creating a false-positive risk. The TSPEC should note this limitation explicitly so engineers know non-ASCII search behavior is not covered by the unit tests in `BearingHistoryViewModelTest.kt`. | §9.2 |
| F-10 | Low | `AT-HIST-01-A` (§9.4) counts badge-visible rows using a per-position loop with a `try/catch(NoMatchingViewException)` heuristic. This approach counts any view with `R.id.interference_badge` that is `isDisplayed()`, which is true for views that are on-screen. For positions 5–9 (off-screen on a typical device), the assertion pattern may not scroll the RecyclerView to bring each item into display before checking. The test calls `RecyclerViewActions.scrollToPosition` before each check, which is correct, but the test body shown in §9.4 does not include the `scrollToPosition` call in the loop — only the badge check. The missing scroll call means off-screen items will not be checked, and the test will pass even if badges are incorrectly absent on items 5–9. | §9.4 |

## Questions

| ID | Question |
|----|---------|
| Q-01 | §5.3 `DriftDetector`: when the timer is in COUNTING and the elapsed time exceeds 60 s but deviation is ≤ 10%, the detector stays COUNTING from the original `countingStartMs`. This means a frame at t=65 s with deviation 9.9% and a subsequent frame at t=66 s with deviation 10.1% will trigger. Is this intended? If deviation fluctuates around the threshold after the 60-second mark, a single above-threshold frame fires `TRIGGERED`. Should there be a minimum number of consecutive above-threshold frames before triggering, or is single-frame sufficient? The FSPEC §FSPEC-CAL-03 says "evaluated only when the timer has elapsed (> 60 s)" but does not specify a persistence requirement for the deviation reading. This is testable either way, but needs to be explicit. |
| Q-02 | §6.1.4 Drift Detection in sensor loop: `expectedFieldUt` is cached in a ViewModel field and updated only on `loadCalibrationAge()` calls. The TSPEC documents that "Float writes are atomic on JVM" and this is "safe." However, `loadCalibrationAge()` writes `expectedFieldUt` from `Dispatchers.IO` while the sensor loop reads it from `Dispatchers.Default`. While JVM Float writes are atomic at the memory level, the visibility guarantee requires `@Volatile` or a similar mechanism to ensure the Default-dispatcher thread sees the updated value without a stale cache. The existing code does not show `@Volatile`. Is this an accepted risk, or should the field be marked `@Volatile`? If accepted, the TSPEC should explicitly document why. |
| Q-03 | `AT-CAL-02-C` (§9.6) asserts `driftDetector.elapsedCountingMs() == null` after `resetDriftDetector()`. But `AT-CAL-02-C` in the FSPEC is an instrumented test ("Instrumented test, FakeClock, fake DriftDetector stub"). The TSPEC assigns the `elapsedCountingMs()` assertion to a test in `BearingHistoryViewModelTest.kt` — which is a unit test file. Should `AT-CAL-02-C` be split into: (a) an instrumented test for the banner GONE + cooldown cleared assertions, and (b) a unit test for the timer-reset assertion? Or is the full `AT-CAL-02-C` being moved entirely to unit test level? The FSPEC and TSPEC assignment are inconsistent. |

## Positive Observations

- The FakeBearingDao pattern (§9.2) is well-designed as a protocol-based recording fake, consistent with `FakeBearingCaptureUseCase`. Invocation-count tracking via `searchFlowCalls` and `getAllFlowCallCount` is exactly what is needed to test debounce behavior without relying on timing heuristics.
- The two-launcher design for `ActivityResultLauncher` (§6.2) eliminates ambiguity in the `RESULT_OK` handler and makes each result path independently testable — a clear win for testability.
- `DriftDetector.elapsedCountingMs()` exposed as `internal` (§5.3, §9.6) is the right call: it avoids reflection-based testing while keeping the internal state inspectable in tests. Consistent with the project's `@VisibleForTesting` precedent in `CalibrationEngine`.
- The `AT-CAL-03-B2` boundary test at exactly 10% deviation (§9.7) correctly uses `>` semantics. This is a common off-by-one that is explicitly captured.
- The `AccelerometerVarianceTracker` time-based window design (§5.2) is a proactive improvement over the FSPEC's sample-count assumption, and the rationale is well-documented.
- The coverage matrix (§9.1) is the clearest component-level test map seen in this feature's documents. The explicit assignment of each FSPEC acceptance test to a file and test level is implementer-friendly.
- The explicit callout that `AT-HIST-03-D` is a manual test with a documented reason (§3.1, FSPEC-HIST-03 AT-HIST-03-D) and an automated approximation added (`AT-HIST-03-E`) is the correct approach for a process-death scenario.
- The `SensorFileWriter` interface (§5.4) gives the write path a clean seam for unit testing without requiring Robolectric or a device. The design is testability-first.

## Recommendation

**Need Attention**

> F-01 through F-04 are Medium severity. F-01 (missing formula-assertion test name) and F-02 (missing ViewModel unit test for the `TRIGGERED → cooldown-check → VISIBLE` wiring) must be resolved before implementation begins: they cover logic that cannot be caught by any other test in the plan. F-03 (undocumented error-path test status) and F-04 (integration test false-positive risk from early-exit pattern) must also be addressed. Low findings (F-05 through F-10) should be resolved but do not block the start of implementation if Medium findings are addressed first.
