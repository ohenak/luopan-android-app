## Positive Observations

- The separation of pure-function format tests (PROP-HIST-053, PROP-HIST-054) from E2E rendering tests is architecturally correct and will provide fast, stable Unit coverage independent of Espresso flakiness.
- The DriftDetector state machine properties (PROP-CAL-010 through PROP-CAL-016) are precise and cover all boundary conditions — the strictly-greater-than threshold (PROP-CAL-012), the zero-field guard (PROP-CAL-013), and the post-trigger reset semantics (PROP-CAL-015) are all explicit. This is high-quality test design.
- PROP-HIST-044 (second-swipe cancels first undo) and PROP-HIST-045 (at-most-one pending undo) together form a correct pair: the Unit property verifies the ViewModel contract, the E2E property verifies the UI consequence. This decomposition is sound.
- Using `IDriftDetector` with `FakeDriftDetector` injection (PROP-CAL-037) is architecturally correct and consistent with the existing codebase pattern (`FakeBearingCaptureUseCase`, `FakeMagneticFieldModel`).
- The population-variance specification (PROP-CAL-035) is precise: providing the exact expected value (≈ 0.6667) and distinguishing it from sample variance (1.0) eliminates ambiguity about which formula to implement.
- Cross-review findings from TE and PM are cleanly locked down as named properties (PROP-HIST-041, PROP-CAL-016, PROP-CAL-036, PROP-CAL-037) with explicit pass conditions. This is good practice.
- Section 6.3 (Deliberately Out-of-Scope) is well-structured — documenting what is not tested and why prevents future reviewers from filing false gaps.

## Recommendation

**Need Attention**

> Seven High or Medium findings are present. The blocking issues before implementation can start are:
>
> 1. **F-01**: Add `espresso-contrib` to `libs.versions.toml` and `app/build.gradle.kts` before any Phase F instrumented tests are written.
> 2. **F-02**: Specify that `BearingDaoTest.kt` Flow tests use `runTest` with a `TestScope`, not `runBlocking`, and confirm `TestBearingDatabase` is upgraded to match the new DAO version.
> 3. **F-10**: Rewrite `PROP-SENSOR-037` pass condition to use a `SensorInfo` wrapper type (or equivalent abstraction) instead of `List<android.hardware.Sensor>` to enable JVM testability.
> 4. **F-08**: Acknowledge that `PROP-CAL-038` requires Robolectric (not plain JVM) due to `AndroidViewModel`, or refactor the integration wiring into an Android-free component.
> 5. **F-06**: Replace the spy-based pass condition in `PROP-HIST-022` with a recording adapter approach (no mocking library is available).
