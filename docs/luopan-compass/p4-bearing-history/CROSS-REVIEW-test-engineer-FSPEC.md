| F-01 | High | **AT-HIST-02-B asserts "DAO query invoked exactly once" but the filtering strategy is unresolved — this assertion is only valid for one of the two implementations.** The FSPEC describes search filtering as either "client-side (in-memory)" or "via SQL `LIKE '%query%'`" without committing to one. If filtering is client-side on an already-subscribed `Flow<List<BearingRecord>>`, the DAO is never additionally invoked during a search — asserting "DAO query invoked exactly once" will fail. If filtering uses a new parameterised Room query, the assertion is correct. A test engineer cannot write AT-HIST-02-B without knowing which path is taken. The FSPEC must commit to one filtering strategy; the acceptance test must be updated to match. A compatible alternative: assert "UI list shows exactly 2 matching items" rather than counting DAO invocations, which is strategy-agnostic and automatable regardless of implementation path. | FSPEC-HIST-02 AT-HIST-02-B; Behavioral Flow §3; Business Rules "No network or disk I/O delay" |
| F-02 | High | **AT-HIST-03-D (process-death no-undo) is not automatable as specified.** The test asserts "process is terminated" while a Snackbar is visible, then relaunches and verifies no undo is offered. Reliably terminating an instrumented test process mid-run without destroying the test infrastructure itself requires `UiAutomation.executeShellCommand("am kill <pid>")` or `ActivityScenario` + `ProcessPhoenix` patterns — none of which are mentioned. More critically, a standard instrumented test runner will report a test failure if the process under test is killed externally. The FSPEC must specify the test harness: either (a) an instrumented test that uses `ActivityScenario.recreate()` to simulate lifecycle death (ViewModel is destroyed, but DB state survives), which is the closest automatable approximation; or (b) explicitly classify this as a manual test scenario with a note that process kill cannot be deterministically asserted in a standard Espresso/JUnit4 test. Leaving it as "Instrumented test" with "process is terminated" is not implementable. | FSPEC-HIST-03 AT-HIST-03-D |
| F-03 | High | **AT-CAL-02-A does not test the ViewModel layer — it tests a "fake DriftDetector" emitting directly into the ViewModel, bypassing the actual `DriftDetector`→ViewModel integration.** The test is classified "Unit test (FakeClock, fake DriftDetector)" and its *When* is "DriftDetector emits TRIGGERED." Without this integration test, it is possible to have a ViewModel that responds correctly to a mocked event, and a `DriftDetector` that correctly emits events in isolation, but with a broken wiring between them that no test would catch. An integration test at the ViewModel level — using a real `DriftDetector` with `FakeClock`, driven by injected sensor frames — is required to cover this boundary. | FSPEC-CAL-02 AT-CAL-02-A; FSPEC-CAL-03 AT-CAL-03-B |

Each mechanism has different lifecycle implications for Fragment state. If the ViewModel retains `calAgeBannerDismissed = true` across Fragment recreation but not across Activity recreation, the test could give a false pass or false fail depending on which mechanism is used. The test must specify the navigation mechanism precisely. | FSPEC-CAL-01 AT-CAL-01-D; AT-CAL-01-E |

True process death requires re-launching a new `ActivityScenario`, which does destroy the ViewModel. The test must specify: "A new `ActivityScenario<CompassActivity>` is launched with the same FakeClock and SharedPreferences state," and that the prior scenario was closed. This is a real difference in test behavior, not just wording. | FSPEC-CAL-01 AT-CAL-01-E |

Without a specified assertion mechanism, two engineers will write incompatible tests. A concrete assertion: "the text of `captured_at` displayed in row 0 is later than the text in row 1, which is later than row 2, etc." or "the RecyclerView adapter item at index 0 has `captured_at` == the maximum `captured_at` in the seeded dataset." | FSPEC-HIST-01 AT-HIST-01-A |

A regression test is needed: type "n" at t=0, type "o" at t=250 ms (before 300 ms elapses), assert no query fires at t=300 ms (the timer must have restarted from t=250 ms), then assert query fires at t=560 ms (250 + 300 + margin). ## Positive Observations

- **DriftDetector state machine (FSPEC-CAL-03) is exceptionally testable.** All five acceptance tests (AT-CAL-03-A through AT-CAL-03-E) use `FakeClock` with concrete millisecond values, specific `expected_field_ut` inputs, and unambiguous `DriftEvent` output assertions. The no-hysteresis rule (AT-CAL-03-E) is precisely specified with a concrete variance value above threshold. These tests can be implemented without further clarification.
- **Condition A and B dismiss mechanics are fully specced as separate, testable branches.** The dismiss table in FSPEC-CAL-01 and FSPEC-CAL-02 enumerates three affordances per banner with distinct `Immediate Effect` and `Persistence` columns. Each row maps directly to a separate test case, and the acceptance tests cover all three paths for Condition A (AT-CAL-01-D, AT-CAL-01-E, AT-CAL-01-F).
- **The floor-division rule for day-count N is isolated into its own unit test (AT-CAL-01-C).** Extracting this into a pure function test (`computeCalibrationAgeDays(elapsedMs)`) rather than embedding it inside a larger fragment test is the right testability decision. Boundary values (31d 23h → 31) are concrete and automatable.
- **Field deviation and inclination deviation format rules (FSPEC-HIST-05) are stated with worked examples that are directly automatable.** AT-HIST-05-D (negative inclination: `-2.3` → `"-2°"`) correctly targets the truncation-toward-zero edge case, which differs from floor for negatives. This is a common display bug and is well-specified.
- **FSPEC-CAL-03-D explicitly tests the `expected_field_ut = 0.0` disabled state**, which guards against a division-by-zero. The test is a pure unit test with zero Android dependencies. This is correctly placed at unit level.
- **AT-SENSOR-01-B's "timestamp same as prior launch" assertion is a strong correctness signal** — not just checking file existence but asserting the file was not re-serialized. This is implementable by comparing the `written_at_iso8601` field value between two runs.
- **The two-banner simultaneous display test (AT-CAL-01-G) is present** and correctly classified as an instrumented test. Specifying "distinct, non-merged UI elements" gives the test engineer a concrete structural assertion (two separate view elements are visible) rather than a vague "both visible" check.
- **The `Clock` interface dependency is correctly specified in FSPEC-CAL-03** with the production injection path (`CompassViewModel.Factory`) and the test injection path (`FakeClock`), referencing the correct precedent file (`CompassViewModelTest`). This eliminates the most common testability antipattern for time-dependent code.

## Recommendation

**Need Attention**

> Three High findings are present (F-01, F-02, F-03), each of which blocks test authoring for a different domain.
>
> 1. **F-01 (mandatory — AT-HIST-02-B cannot be implemented):** Commit to a single search-filtering strategy (client-side in-memory vs. SQL `LIKE`). Replace the "DAO invoked exactly once" assertion in AT-HIST-02-B with a strategy-agnostic UI-observable assertion (e.g., "emitted list contains exactly N items matching the query") that works regardless of implementation path.
>
> 2. **F-02 (mandatory — AT-HIST-03-D is not automatable as written):** Either reclassify the process-death scenario as a manual test with an explanatory note, or specify the test harness explicitly (e.g., "close the ActivityScenario and launch a fresh one; the ViewModel is destroyed; verify DB state on relaunch").
>
> 3. **F-03 (mandatory — ViewModel↔DriftDetector integration boundary is untested):** Add an integration test that wires a real `DriftDetector` (with `FakeClock`) into a real `CompassViewModel` and drives it with injected sensor frames through the full 60-second window, asserting that `driftBannerVisible` becomes `true` in the ViewModel's state. This cannot be omitted — it is the only test that would catch a wiring bug between the two components.
>
> 4. **F-04, F-05 (Medium — lifecycle mechanism must be specified before test authoring):** AT-CAL-01-D and AT-CAL-01-E must specify the navigation/recreation mechanism used to simulate session persistence and process death, respectively.
>
> 5. **F-06, F-07, F-08 (Medium — missing coverage):** Add an assertion mechanism for sort-order in AT-HIST-01-A; add a debounce-restart unit test (keystroke at 250 ms must not fire the search); add a post-trigger IDLE-reset test for `DriftDetector`.
>
> 6. **F-09 through F-15 (Low — specify or document exceptions):** Clarify expand/collapse assertion mechanism (F-09); specify "row expanded" trigger in AT-HIST-05-C (F-10); tighten AT-HIST-03-C timing (F-11); define injectable abstraction for AT-SENSOR-01-E (F-12); correct "mock InterferenceState" language (F-13); either add or explicitly exempt the scroll-on-undo AT (F-14); fix the tab-absence assertion in AT-CAL-02-F (F-15).
