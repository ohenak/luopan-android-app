# Cross-Review: TE — REQ-luopan-p2-true-north-capture

**Reviewer role:** Senior Test Engineer
**Document reviewed:** REQ-luopan-p2-true-north-capture.md
**Date:** 2026-04-24
**Iteration:** 1

---

## Summary

The Phase 2 REQ is well-scoped and generally readable, but several acceptance criteria in §8 contain ambiguities that would prevent an engineer from writing deterministic tests without additional decisions. The most critical gaps are: (1) Scenario C states "15 days ago" as a concrete value but provides no expected UI string format to assert against; (2) REQ-CAPTURE-04 names encryption as a requirement but the phase REQ omits any verifiability criterion for the bearing database, unlike Phase 1's `PROP-PERSIST-02` for calibration; (3) REQ-NFR-08 cold-start measurement methodology is underspecified — "app icon tap" to "first valid heading" has at least four plausible interpretations and no reference device is named. Together these produce five High findings and five Medium findings. The document cannot proceed to FSPEC authoring without resolving the High findings.

## Questions

| ID | Question |
|----|---------|
| Q-01 | Scenario C specifies "15 days ago" as the cache age. Will the test inject this age via a `FakeClock` or by actually writing a timestamped record 15 days in the past? If via `FakeClock`, the age-formatting function must accept an injectable time source — is this intended? |
| Q-02 | REQ-NORTH-02 specifies a fallback label "Android model — may be less accurate." Is this an exact string or a description? If exact, it must be codified in a string resource and referenced by resource ID in the acceptance criterion. |
| Q-03 | Scenario E mentions "Capturing with confidence 'Poor': bearing saves with `interference_flag=true`." Is `interference_flag` set purely based on `InterferenceState.WARNING` at capture time, or also when confidence is POOR due to other reasons (e.g., tilt, calibration age)? The mapping from confidence level to `interference_flag` is not specified. |
| Q-04 | Risk P2-R3 raises the 30-day cache cutoff as an assumption that may need to be 7 days for high-secular-variation areas. If the cutoff becomes location-dependent, how should tests handle this? Is there a per-location configurable threshold, or a single global constant? |
| Q-05 | REQ-NORTH-04 mentions "Grid N" is deferred to Phase 5. Should Scenario E (bearing capture) explicitly assert that `north_type` CANNOT be GRID in Phase 2, and that the UI does not offer GRID as a capture option? This is a negative constraint that should be testable. |

## Positive Observations

- The WMM2025 offline-first design (REQ-NORTH-01, Scenario B) is well-specified with a concrete ±0.1° tolerance and a "no network request" assertion — these are testable at the integration level.
- Scenario D covers the no-GPS/no-cache path with clear branching: manual entry OR dismiss to Magnetic North only. Both branches have distinct expected outcomes.
- The interference detection upgrade (Scenario F, §5.1 conformance note) is correctly scoped as a conformance upgrade of Phase 1 requirements rather than a new requirement ID, avoiding traceability fragmentation.
- REQ-CAPTURE-06 location privacy toggle defaulting to ON with per-capture (non-persistent) behavior is explicitly stated — this is a testable property that would otherwise be easy to leave implicit.
- The deferral table (§6) is precise and includes requirement IDs, reducing the risk of accidentally implementing or testing out-of-scope items.
- Phase 1's PROPERTIES pattern (PROP-PERSIST-02 for calibration DB encryption) is a strong precedent that should be directly replicated for the bearing DB in Phase 2.

---

## Recommendation: Need Attention

Five High findings must be resolved before FSPEC authoring begins. TE-REQ-01 through TE-REQ-05 each block the ability to write a deterministic, non-flaky automated test for the affected scenario or requirement. The Medium findings (TE-REQ-06 through TE-REQ-10) should be resolved during FSPEC authoring at the latest, as they will otherwise propagate into the TSPEC as ambiguous test cases.
