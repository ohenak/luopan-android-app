## Summary

The PROPERTIES document defines 36 machine-checkable properties across 9 domains and is well-structured. All 9 in-scope REQ IDs (REQ-NORTH-01 through REQ-NORTH-04, REQ-DECL-01, REQ-DECL-02, REQ-CAPTURE-01, REQ-CAPTURE-02, REQ-CAPTURE-04, REQ-CAPTURE-06, REQ-NFR-08) appear in the coverage matrix. All 6 REQ §8 acceptance scenarios (A–F) have traceable property coverage. Priorities are consistently aligned with REQ designations. No Phase 3/4/5 behavior is introduced.

Four issues require attention before the document can be approved:

1. **Schema type conflict (P0 blocker):** PROP-SCHEMA-01 declares `id` as `Long` (autogenerated primary key), contradicting FSPEC §6.1 which canonically defines `id` as `String` (UUID v4, `TEXT` column). This mismatch would produce a failing test on a correct implementation.
2. **Schema field name mismatch (P0 blocker):** PROP-SCHEMA-01 uses `altitude_m` for the Kotlin field name, but FSPEC §6.1 uses `alt_m` as the canonical column name with an explicit normative note that `altitude_m` is the REQ name mapping to column `alt_m`. A reflection-based test asserting `altitude_m` will fail against the canonical implementation.
3. **FSPEC AT-G not tagged in FSPEC column (Minor):** The FSPEC extends REQ §8 with AT-G (9 additional scenarios: extreme latitude, WMM expiry, cache boundary, GRID UI guard, GPS-denied full flow). Properties covering these scenarios exist but are not labeled with AT-G in their FSPEC column, making traceability harder to verify at a glance.
4. **FSPEC panel fields partially covered (Minor):** PROP-DECL-02 verifies 4 fields matching REQ-DECL-02. FSPEC §2.3 adds 3 further required display fields (coordinates type label, cache age "N days ago", and the "True North is currently off" note when Magnetic N is active) that have no corresponding property.

One additional observation does not block approval but is flagged for completeness:

- REQ-CAPTURE-06 requires the toggle to default to ON. No property verifies this default state.

## Recommendation: Approved with Minor Issues

The document is structurally sound and provides strong P0 coverage across all in-scope REQ requirements and REQ §8 acceptance scenarios. Items PM-P-01 and PM-P-02 are schema-level type and name conflicts that would cause the test suite to either fail spuriously or silently miss real defects; they must be resolved before test implementation begins. Items PM-P-03 through PM-P-05 are traceability and coverage gaps that do not block test writing but should be addressed in the same revision pass. The document is approved for implementation once PM-P-01 and PM-P-02 are corrected.
